# Исполнительное резюме

Проект ZaimKarta — это **витрина микрозаймов и МФО** с монетизацией через CPA‑сети. Чтобы запустить сервис в продакшн, нужно одновременно обеспечить стабильную работу сайта, безопасность пользовательских данных и соблюдение SEO‑требований. При ограниченном бюджете и отсутствии отдельной DevOps‑команды рекомендуется стартовать на **управляемом VPS со сборкой через Docker**. Такой вариант предлагает приемлемый баланс между ценой, удобством и контролем. Предлагаемый план охватывает выбор инфраструктуры, настройку домена и HTTPS, деплой Next.js‑приложения и PostgreSQL, защиту сервера и админки, организацию резервного копирования, мониторинг, подготовку к SEO‑индексации и юридические требования.

### Почему выбран VPS с Docker и Caddy

* **Контроль и предсказуемость**. Собственный VPS даёт root‑доступ, гибкое управление конфигурацией и отсутствие рисков санкций/блокировок. Российские хостинги (например, *Beget*) предлагают VPS в России, Европе и Казахстане, готовые за 10 секунд и резервное копирование по умолчанию【164026651930349†L46-L119】.
* **Простота развертывания**. Docker стандартизирует окружение и позволяет легко запускать несколько сервисов (Next.js, PostgreSQL, Caddy) на одной машине. Caddy автоматически получает и обновляет сертификаты Let's Encrypt【720866858224123†L219-L263】, что снижает нагрузку на владельца.
* **Масштабирование**. Для старта достаточно 1–2 vCPU, 2–4 ГБ RAM и 30 ГБ NVMe (Beget предлагает тариф 2 vCPU, 2 ГБ, 30 ГБ за ~27 ₽/день【164026651930349†L82-L119】). В будущем можно перейти на отдельный managed‑PostgreSQL и вынести приложение в PaaS‑платформу.

## Пошаговый план деплоя

### 1. Подготовительные шаги

1. **Зарегистрировать домен** `zaimkarta.ru` у российского регистратора.
2. **Выбрать хостинг** (рекомендуется Beget) и заказать VPS (2 vCPU, 2 GB RAM, 30 GB NVMe). Сразу включить автоматические снапшоты.
3. **Настроить DNS**:
   * Создать A‑запись `zaimkarta.ru` → IP VPS.
   * Добавить A‑запись `www.zaimkarta.ru` → IP VPS и настроить редирект (основным адресом выбрать `https://zaimkarta.ru`, а `www` переадресовывать).
   * Создать записи `MX` (если нужна почта) и `TXT` для SPF, DKIM, DMARC. Для подтверждения прав в Яндекс Webmaster добавить TXT‑запись с уникальным кодом (один из методов верификации【76917126251400†L126-L170】).
4. **Подготовить репозиторий** (GitHub/GitLab). Создать `.env.example`, добавить `.env` в `.gitignore` и хранить переменные окружения только на сервере【709411522399450†L22-L24】.
5. **Собрать Docker‑образ** для Next.js (использовать официальный `node:18-alpine`), установить зависимости, собрать проект (`npm install && npm run build`), затем запустить команду `npm start`. Для PostgreSQL использовать официальный образ `postgres:15-alpine`.

### 2. Настройка сервера и деплой

1. **Установить Docker и docker‑compose** на VPS.
2. **Создать docker‑compose.yml** с тремя сервисами: `caddy` (или `nginx`), `app` (Next.js), `db` (PostgreSQL). Определить том для постоянного хранения БД (`db_data`). Пример:

   ```yaml
   version: '3.9'
   services:
     db:
       image: postgres:15-alpine
       environment:
         POSTGRES_USER: zaimkarta
         POSTGRES_PASSWORD: <генерировать>
         POSTGRES_DB: zaimkarta
       volumes:
         - db_data:/var/lib/postgresql/data
       networks:
         - backend
     app:
       build: .
       environment:
         DATABASE_URL: postgresql://zaimkarta:<пароль>@db:5432/zaimkarta
         NODE_ENV: production
       depends_on:
         - db
       networks:
         - backend
     caddy:
       image: caddy:alpine
       ports:
         - "80:80"
         - "443:443"
       volumes:
         - ./Caddyfile:/etc/caddy/Caddyfile
         - caddy_data:/data
       networks:
         - backend
   volumes:
     db_data:
     caddy_data:
   networks:
     backend:
   ```

3. **Caddyfile**: настроить HTTP→HTTPS и проксирование на контейнер `app`. Пример:

   ```Caddyfile
   zaimkarta.ru {
     reverse_proxy app:3000
     encode gzip
     log
   }
   www.zaimkarta.ru {
     redir https://zaimkarta.ru{uri}
   }
   ```

   При первом запуске Caddy автоматически получит сертификат Let’s Encrypt и настроит перенаправления【720866858224123†L219-L263】.

4. **Запуск**: `docker-compose up -d` развернёт базу, приложение и прокси. После первого запуска выполнить `docker exec app npm run prisma migrate deploy` для применения миграций.
5. **Настроить резервные копии**: использовать cron в отдельном контейнере или host‑системе, который ежедневно делает `pg_dump` и копирует архивы на удалённый сервер/S3.
6. **Настроить окружение**: заполнить переменные `DATABASE_URL`, секреты JWT, API‑ключи CPA‑сетей. Использовать менеджер паролей (Bitwarden, 1Password) и ограничить доступ к секретам.
7. **Настроить CI/CD**: подключить репозиторий к GitHub Actions или GitLab CI для автоматической сборки образа и деплоя на сервер (через `ssh` или `rsync`). Разделить ветки `staging` и `production`. В staging использовать отдельную базу.

### 3. Раздельное размещение базы

* На старте база может работать в контейнере `db`. Для повышения отказоустойчивости и удобства бэкапов в будущем перенести её в managed‑сервис (например, Selectel Managed PostgreSQL). Управляемые сервисы автоматически создают реплики, делают point‑in‑time‑бэкапы и избавляют от администрирования.
* При переносе важно обновить `DATABASE_URL` и обеспечить доступ только с IP сервера (правила firewall). Удалённое подключение к базе через публичный интернет нужно закрыть: оставить `listen_addresses = 'localhost'`【732434840283133†L39-L56】 и настроить `pg_hba.conf` на разрешение соединений только с нужного хоста【928933687495566†L64-L124】.

### 4. Стратегия отката и staging

* Развернуть **staging‑окружение** (например, поддомен `staging.zaimkarta.ru`) на том же сервере или отдельном VPS. Использовать отдельные контейнеры и базу.
* При деплое сначала отправлять обновления на staging, выполнять миграции, проверять работу. Только затем деплоить на production.
* Для отката хранить предыдущие Docker‑образы; при неудаче переключать обратный прокси на предыдущий контейнер.

## План для базы данных PostgreSQL

1. **Создание пользователя и базы**: создавать отдельного пользователя с минимальными правами; использовать сложный пароль. В `pg_hba.conf` разрешить подключение только для этого пользователя и только по внутреннему адресу.
2. **Ограничение доступа**: не открывать порт 5432 наружу; оставить `listen_addresses` по умолчанию (`localhost`)【732434840283133†L39-L56】. Настроить firewall (`ufw`/`iptables`) для разрешения доступов только контейнерам приложения.
3. **Миграции**: управлять схемой через Prisma (`npx prisma migrate deploy`), соблюдать порядок миграций, хранить их в репозитории.
4. **Бэкапы**:
   * Делать ежедневные полные дампы (`pg_dump`) и хранить их в зашифрованном виде. Соблюдать правило 3‑2‑1: три копии, две разные среды, одна копия вне площадки【601496682850598†L90-L108】.
   * Хранить бэкапы не менее 7 дней, создавать недельные и месячные архивы. Использовать сжатие (gzip, lz4) для экономии места【601496682850598†L146-L168】.
   * Включить WAL‑архивацию для point‑in‑time recovery: сохранять журнал транзакций и регулярно отправлять его в хранилище【601496682850598†L233-L258】.
   * Регулярно проверять восстанавливаемость — еженедельный тест восстановления из бэкапа【601496682850598†L206-L225】.
5. **Шифрование**: хранить дампы зашифрованными (например, с `gpg`) и передавать их по зашифрованному каналу (SFTP, HTTPS). Удалять старые дампы согласно политике retention.

## Security‑checklist

Следующий контрольный список помогает закрыть основные уязвимости:

* **HTTPS‑только**: перенаправлять все запросы на HTTPS, использовать HSTS с длинным сроком (например, 1 год)【411940888940075†L143-L176】.
* **Secure‑headers**: настроить Content‑Security‑Policy, X‑Content‑Type‑Options (`nosniff`), Strict‑Transport‑Security, Referrer‑Policy, Permissions‑Policy, X‑Frame‑Options и cross‑origin‑headers. Эти заголовки уменьшают риск XSS, MIME‑sniffing и clickjacking【411940888940075†L143-L176】.
* **Защита админки**: отдельный URL/поддомен, использование 2FA, сложные пароли, ограничение попыток входа и IP‑whitelist. Отключить индексацию `/admin` (robots.txt, meta noindex). Вести audit‑лог действий админов.
* **Авторизация и аутентификация**: использовать проверенные библиотеки (NextAuth.js, Passport). Пароли хешировать через bcrypt/argon2; хранить солёные хеши. Для работы админки предусмотреть roles & permissions.
* **Rate limiting**: на уровне API внедрить ограничение количества запросов (express‑rate‑limit). Например, максимум 100 запросов за 15 минут; превышение выдаёт HTTP 429【766137314707241†L60-L97】.
* **CSRF‑защита**: генерировать CSRF‑токены и проверять их в формах; для cookies использовать атрибуты `SameSite=Lax`/`Strict`【956950088603007†L355-L383】. Cookies должны быть `HttpOnly` и `Secure`【493224933279896†L310-L401】.
* **XSS‑защита**: везде экранировать пользовательский ввод, использовать `helmet`/`next-secure-headers`, избегать небезопасного `dangerouslySetInnerHTML`.
* **SQL‑инъекции**: использовать ORM Prisma (prepared statements), валидацию входных данных.
* **CORS‑политика**: разрешать запросы только с нужных доменов.
* **Обработка ошибок**: не выводить stack‑trace в продакшн; логировать ошибки в Sentry или другом мониторе.
* **Запрет индексации**: robots.txt должен запрещать `/admin`, `/api/*`, `/go/*` (редиректы на партнёров), страницы ошибок.
* **Защита файловых загрузок**: проверять типы и размер файлов; сохранять в изолированном хранилище; запрещать выполнение загруженных файлов.

### Риски публичных частей

* **SEO‑страницы**: риск XSS и медленной загрузки. Нужно оптимизировать контент, использовать lazy‑loading и проверять параметры.
* **Эндпоинт переходов `/go/...`**: может стать целью накрутки кликов. Вводить rate limiting, фильтрацию ботов (User‑Agent, IP‑репутация), хранить минимум персональных данных (IP, user‑agent, click_id). CPA‑параметры передавать через сервер, избегая вывода секретных токенов.
* **Формы обратной связи**: проверять данные на сервере, использовать reCAPTCHA/Yandex Captcha, защищать от спама.
* **Загрузка логотипов МФО**: проверять MIME‑тип, ограничивать размер, хранить файлы отдельно от кода (S3/облачный storage).

## Защита эндпоинта партнёрских переходов `/go/...`

* **Логирование**: сохранять `click_id`, timestamp, IP, user‑agent и идентификатор МФО. Не хранить персональные данные, такие как cookie пользователя.
* **Фильтрация ботов**: проверять `User‑Agent`, использовать списки подозрительных IP, возможно интегрировать reCAPTCHA/поведенческий анализ. Ограничить число переходов с одного IP за определённое время.
* **Rate limiting**: применять отдельный лимит для переходов (например, 30 кликов/минуту с IP). При превышении записывать событие и не засчитывать клик.
* **Параметры CPA**: хранить партнёрские токены на сервере; передавать пользователю только параметры, разрешённые сетью. Использовать `rel="sponsored"` или `nofollow` для всех партнёрских ссылок, чтобы поисковики не передавали вес【417274642434045†L415-L431】.
* **SEO и пользовательский опыт**: не индексировать `/go` в robots.txt; перенаправлять со статусом `302` (временный), чтобы не кэшировать редиректы. Стараться, чтобы редирект происходил быстро (< 100 мс).

## Мониторинг и алерты

* **Доступность сайта**: подключить UptimeRobot (бесплатный план включает 50 мониторов, 5‑минутный интервал и до 20 интеграций【216708624483481†L168-L190】). Настроить проверки HTTP(S) главной страницы, стороничных SEO‑страниц и `/go`‑редиректа. Установить уведомления в Telegram/email.
* **Ошибки и производительность**: интегрировать Sentry. Он захватывает исключения, производительность, session replay и выводит стек‑трейсы с данными пользователя【223149248568464†L56-L60】. Бесплатный тариф даёт 5 000 ошибок/мес. Настроить уведомления в Slack/Telegram.
* **Серверные метрики**: настроить мониторинг CPU, RAM, диска и состояния контейнеров. Можно использовать простую связку `Node Exporter + Prometheus + Grafana`, но для старта достаточно метрик хостинга и алертов по email.
* **Состояние PostgreSQL**: следить за временем ответа запросов, количеством соединений и ростом объёма базы. Можно использовать `pgAdmin`, `pg_stat_activity` или встроенный мониторинг managed‑сервиса.
* **Бэкапы и SSL**: создать cron‑задачи, которые проверяют наличие свежих бэкапов и срок действия TLS‑сертификатов. UptimeRobot может отслеживать срок действия SSL, но в бесплатном плане это недоступно; проверку можно организовать скриптом.
* **Аномалии кликов**: настроить simple dashboards (например, через Grafana) для отслеживания количества кликов по каждому партнёру. Резкие всплески могут свидетельствовать о накрутке.

### Что обязательно на старте

* Мониторинг доступности (UptimeRobot).
* Sentry (или аналог) для ошибок.
* Логирование запросов и ошибок в файлы/stdout с ротацией (например, через `logrotate`).
* Оповещения в Telegram или email.

### Что можно добавить позже

* Полноценный стек Grafana + Prometheus для метрик и алертов.
* Журналирование действий админов с последующей аналитикой.
* Система реагирования (PagerDuty, Better Uptime).

## Логи

Рекомендуется собирать следующие логи и хранить их не менее 30 дней (кроме персональных данных):

* **Доступ к веб‑серверу** (NGINX/Caddy): IP, timestamp, URL, HTTP‑код, user‑agent.
* **Логи приложения**: ошибки Node.js/Next.js, stack‑trace (без утечки секретов), события входа/выхода.
* **Логи базы данных**: медленные запросы, ошибки подключения, успешные/неуспешные миграции.
* **Логи админки**: кто выполнил действие, какое именно, время.
* **Логи переходов `/go`**: зафиксированные клики, статусы редиректа.
* **Попытки авторизации**: успешные и неудачные, превышение лимита.
* **Системные логи**: CPU/RAM/диск, перезагрузки контейнеров.

Логи можно хранить в `/var/log` или отправлять в централизованный сервис (Grafana Loki, Elasticsearch). Не хранить лишние персональные данные (фамилии, паспортные данные). Периодически очищать/архивировать старые файлы.

## SEO‑подготовка перед выпуском

Сайт должен быть готов к индексации Яндексом и Google. Контрольный список:

1. **robots.txt**: разрешить индексацию основных страниц (`/` и страницы статей), запретить `/admin`, `/api/*`, `/go/*`, страницы авторизации, черновики.
2. **sitemap.xml**: автоматически генерировать и обновлять карту сайта (Next.js может генерировать sitemap при сборке). Указывать приоритет и дату изменения. Загрузить в Яндекс Webmaster и Google Search Console.
3. **Canonical**: для динамических страниц указывать `<link rel="canonical" href="..."/>`, чтобы предотвратить дубли.
4. **Редиректы**: настроить 301‑редиректы с HTTP→HTTPS, `www`→без `www`, старых URL на новые.
5. **404 и 500 страницы**: создать кастомные страницы с полезной информацией, кодом ответа и ссылкой на главную.
6. **Performance**: обеспечить быструю загрузку (LCP < 2,5 с) — оптимизировать изображения, включить gzip/brotli, использовать `next/image`, настроить кэширование, CDN.
7. **Mobile‑friendly**: адаптивная верстка с Tailwind CSS, проверка Lighthouse.
8. **Meta Title/Description**: уникальные заголовки и описания для каждой страницы, отражающие контент и ключевые запросы.
9. **Open Graph / Twitter Cards**: добавить OG‑теги для красивых сниппетов при расшаривании.
10. **Микроразметка**: использовать Schema.org (Article, Breadcrumb) для статей и списков МФО, чтобы улучшить сниппеты.
11. **Внутренняя перелинковка**: связывать статьи и категории, использовать `next/link` для навигации.
12. **Подключение аналитики**: установить счётчик Яндекс Метрики и привязать сайт в Яндекс Webmaster; добавить Google Analytics при необходимости.
13. **Проверка дублей**: в Yandex Webmaster проверить отчёт «дубликаты», исключить параметры сортировки/фильтрации с помощью canonical или robots.txt【76917126251400†L111-L118】.
14. **Affiliate links**: отмечать партнёрские ссылки атрибутом `rel="sponsored"` или `rel="nofollow"`【417274642434045†L415-L431】. Это помогает поисковикам понимать, что ссылки являются платными, и не передавать им «вес».
15. **Закрытие редиректов** `/go`: запретить индексацию, но разрешить переходы для пользователей. Можно вернуть HTTP 200 с `noindex` meta или использовать 302‑редирект.
16. **Регистрация в Яндекс Webmaster**: подтвердить права (HTML‑файл, meta‑тег или TXT‑запись)【76917126251400†L126-L170】, загрузить sitemap, следить за техническими ошибками, duplicate‑страницами и улучшениями.

## Юридические и доверительные элементы

* **Политика конфиденциальности**: описать, какие данные собираются (IP, cookies, данные форм), как они используются и хранятся.
* **Пользовательское соглашение**: условия использования сайта, ограничения ответственности, правила размещения контента.
* **Раскрытие партнёрских ссылок**: указать, что сайт получает вознаграждение от партнёров; пометить такие ссылки `rel="sponsored"`【417274642434045†L415-L431】.
* **Контакты**: указать юридический адрес, email, телефон поддержки, форму обратной связи.

* **Предупреждение о рисках**: разместить предупреждение о финансовой ответственности при получении микрозаймов.

## Резервное копирование и аварийное восстановление

| Сценарий | Действия |
|---|---|
| **Сервер недоступен (железо/VPS упал)** | Перейти на резервный VPS (готовый заранее или создаваемый из snapshot). Восстановить последние бэкапы базы. Обновить DNS/A‑запись на новый IP. |
| **Сломался деплой (ошибка приложения)** | Откатиться на предыдущий Docker‑образ или git‑коммит. Использовать staging для проверки. |
| **Повреждена база** | Остановить приложение, восстановить последнюю пригодную копию (`pg_restore`), применить WAL‑журналы до нужного момента (PITR). |
| **Сайт взломали** | Изолировать сервер, проанализировать логи, сменить все пароли и секреты. Развернуть чистый сервер из бэкапа, обновить зависимости, закрыть уязвимость. |
| **DNS/домен настроены неверно** | Проверить записи DNS, TTL, обновить через регистратор. Отдельно хранить инструкции по настройке домена. |
| **Бэкап оказался нерабочим** | Регулярно тестировать восстановление (автоматические тесты). Хранить инструкции по восстановлению, скрипты и пароли в защищённом репозитории/менеджере. |

Дополнительные рекомендации:

* Использовать автоматизацию бэкапов и настроить отчёты об успешности, потому что автоматизация и регулярные проверки позволяют избежать человеческих ошибок【601496682850598†L118-L144】.
* Применять шифрование и компрессию бэкапов; резервную копию хранить в облаке/отдельном дата‑центре【601496682850598†L146-L168】.
* Хранить инструкции по DR‑плану в недоступном для посторонних месте, но доступном для команды.

## Секреты и доступы

* **Пароли и ключи**: хранить в менеджере паролей (Bitwarden, 1Password). Использовать уникальные длинные пароли и 2FA для всех сервисов.
* **SSH‑ключи**: генерировать на стороне клиента, хранить в защищённой папке, закрывать доступ к паролю root, использовать `ssh-copy-id`.
* **Файлы `.env`**: хранить на сервере вне репозитория, добавлять в `.gitignore`【709411522399450†L22-L24】. Для переменных, которые должны быть доступны в браузере, использовать префикс `NEXT_PUBLIC_`【709411522399450†L139-L167】.
* **Доступы к хостингу и CPA‑сетям**: ограничить количество аккаунтов, назначить минимальные права, использовать 2FA и логировать входы.
* **API‑ключи**: не хранить в коде; загружать в переменные окружения, использовать secrets менеджеры (Vault, AWS Secrets Manager) при возможности.
* **Доступ к БД**: выдавать только приложению; администратор подключается через ssh‑туннель; запрещать внешние подключения.
* **Доступ к Яндекс Метрике и Вебмастеру**: выделять отдельные аккаунты, подключать двухфакторную аутентификацию, ограничивать права.

## Финальная рекомендация

Для запуска ZaimKarta рекомендуется **использовать российский VPS с Docker и Caddy**, разместить приложение Next.js и базу PostgreSQL в отдельных контейнерах и управлять ими через docker‑compose. Такой подход минимизирует зависимость от зарубежных провайдеров, обеспечивает автоматический HTTPS и даёт полный контроль над конфигурацией при низкой стоимости. В дальнейшем, при росте трафика, базу имеет смысл перенести в managed‑сервис (Selectel/Yandex Cloud) и настроить отдельный сервер для приложения или использовать PaaS. Необходимо внимательно следить за безопасностью (HTTPS, secure‑headers, 2FA, rate‑limiting) и регулярно делать бэкапы. Соблюдая приведённые рекомендации, владелец проекта сможет запустить SEO‑витрину микрозаймов, соответствующую требованиям Яндекса и партнёров CPA, с минимальными рисками и хорошим потенциалом для развития.
