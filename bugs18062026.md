Этот файл нужен как стартовая точка для отдельного диалога по багофиксу.

Команда для нового диалога:

```text
Открой PROJECT_CONTEXT.md и bugs18062026.md. Мы начинаем багофикс. Сначала сгруппируй задачи по зонам проекта, проверь актуальное состояние кода, затем предложи порядок исправления и начинай пошагово.
```

## Важное правило для багофикса

Не чинить все хаотично. Сначала нужно сгруппировать задачи по зонам:

1. Терминология и комплаенс-тексты.
2. Структурированные данные офферов в админке.
3. Логика фильтров и SEO-инструментов.
4. CTA и бесполезные якорные кнопки.
5. SEO-админка, сортировка и порядок блоков на главной.
6. Редактируемые тексты офферов.

Перед правками:

- открыть `PROJECT_CONTEXT.md`;
- посмотреть `git status`;
- убедиться, что локальный сервер и база работают;
- не трогать unrelated changes;
- после каждой группы правок запускать `npm run lint`;
- для крупных публичных изменений проверять страницу в браузере.

## Рекомендуемый порядок исправления

### Этап 1. Сквозная терминология

Сначала убрать публичные вхождения узкого термина `МФО`, где проект говорит обо всех кредиторах.

Почему первым:

- это затрагивает карточки, FAQ, дисклеймеры, инструменты, footer;
- это продуктово и комплаенсно важно;
- это не зависит от будущей нормализации полей.

### Этап 2. Нормализация данных оффера

Затем сделать структурированные поля в админке оффера.

Почему вторым:

- чеклист и калькулятор не смогут нормально фильтровать офферы, пока документы, способы получения и требования вводятся свободным текстом.

### Этап 3. Инструменты и фильтры

После нормализации данных:

- исправить чеклист;
- исправить калькулятор;
- убрать ложное ощущение персонального подбора там, где фильтр еще не реализован.

### Этап 4. CTA и навигация

Исправить бесполезные якорные кнопки и заменить часть CTA на полезные действия.

### Этап 5. SEO-админка и главная

Исправить верстку редактора, сортировку SEO-активов, порядок блоков на главной и страницы архива статей/инструментов.

### Этап 6. Редактируемое описание оффера

Добавить или проверить поле для уникального описания оффера и выводить его на публичной странице.

# Баги и задачи

## 1. Подборщик оффера на главной не работает

URL:

```text
http://localhost:3000
```

Суть:

Форма подбора на первом экране выглядит как инструмент, но кнопка `Подобрать` просто якорит пользователя к блоку `Популярные предложения`.

Текущее поведение:

- пользователь выбирает сумму;
- выбирает срок;
- выбирает критерий `Что важно`;
- нажимает `Подобрать`;
- сайт просто скроллит к офферам.

Ожидаемое поведение:

Форма должна реально влиять на выдачу офферов или быть переосмыслена как честный быстрый переход.

Важное продуктовое решение:

Не обещать точный персональный подбор, если сайт не знает реальные индивидуальные условия, которые кредитор даст конкретному человеку.

Варианты решения:

1. Сделать честный фильтр по структурированным полям оффера:
   - сумма;
   - срок;
   - ставка;
   - способы получения;
   - вероятность одобрения.
2. Если полноценного фильтра пока нет, заменить CTA и текст так, чтобы не создавать ожидание персонального подбора.

Зона проекта:

- `site/src/app/page.tsx`;
- возможный будущий компонент фильтра офферов;
- данные офферов из `site/src/lib/offers.ts`.

Критичность: средняя/высокая.

## 2. В карточках и публичных текстах используется узкое слово `МФО`

URL/зоны:

- главная;
- карточки офферов;
- SEO-подборки;
- FAQ;
- нижние дисклеймеры;
- предупреждения в инструментах;
- footer.

Суть:

Проект будет работать не только с МФО. Возможны:

- МКК;
- МФО;
- КПК;
- ломбарды;
- залоговые кредиты;
- P2P-сервисы;
- другие кредитные платформы.

Поэтому публичное слово `МФО` во многих местах некорректно.

Примеры текущих фраз:

```text
Решение о выдаче принимает МФО.
ZaimKarta показывает выбранные предложения МФО.
Решение принимает конкретная МФО.
Проверять в договоре конкретной МФО.
```

Ожидаемое поведение:

Пройти по всему публичному интерфейсу и заменить `МФО` на более гибкие формулировки.

Базовое правило замены:

- короткие UI-дисклеймеры: `кредитор`;
- поясняющие тексты: `кредиторы`, `партнеры`, `организации, предоставляющие заем`;
- если речь о переходе по CPA: `партнер` или `сайт партнера`;
- если речь о решении по заявке: `кредитор`.

Рекомендуемая основная формулировка:

```text
Решение по заявке принимает кредитор. Перед оформлением проверьте полную стоимость займа и условия договора.
```

Зоны поиска:

```bash
rg -n "МФО|мфо" site/src site/prisma docs
```

Ожидаемые файлы:

- `site/src/components/offer-card.tsx`;
- `site/src/app/[slug]/page.tsx`;
- `site/src/app/offers/[slug]/page.tsx`;
- `site/src/components/seo-tools/*`;
- `site/src/components/site-footer.tsx`;
- seed/demo data в `site/prisma/seed.ts`;
- возможно тексты в базе через seed.

Критичность: средняя/высокая.

```text

```

Технические требования:

- ссылка должна иметь `target="_blank"`;
- добавить безопасный `rel="noopener noreferrer"`;

Зона проекта:

## 5. Чеклист перед заявкой работает как декоративный блок, а не как фильтр

URL:

```text
http://localhost:3000/chek-list-pered-zayavkoy-na-zaym
```

Суть:

Чеклист должен помогать отфильтровать офферы, но сейчас выглядит как условный тест. Как бы пользователь ни отвечал, предложения остаются одинаковыми.

Проблема 1:

Вариант ответа:

```text
Не уверен / примерно / постараюсь
```

не подходит. Это чеклист, а не психологический квиз.

Ожидаемые ответы:

- `Да`;
- `Нет`.

Проблема 2:

Ответы не влияют на список офферов.

Ожидаемая логика фильтрации:

### Паспорт под рукой?

- Если `Нет` → показывать офферы, где паспорт не нужен.
- Если `Да` → показывать офферы, где паспорт является обязательным документом.

Для этого нужны структурированные документы оффера.

### Есть именная банковская карта?

- Если `Нет` → не показывать офферы, где выдача возможна только на карту/СБП.
- Если `Да` → показывать все подходящие офферы.

Для этого нужны структурированные способы получения.

### Понимает, из каких средств будет гасить?

- Если `Да` → показывать все подходящие офферы.
- Если `Нет` → показывать только офферы с высокой вероятностью одобрения.

### Рассчитал переплату?

- Если `Да` → показывать все подходящие офферы.
- Если `Нет` → показывать только офферы с высокой вероятностью одобрения.

### Готов читать договор?

- Если `Да` → показывать все подходящие офферы.
- Если `Нет` → показывать только офферы с высокой вероятностью одобрения.

Зависимость:

Эту задачу лучше чинить после задачи 7, потому что нужны нормализованные поля оффера.

Зона проекта:

- `site/src/components/seo-tools/application-checklist.tsx`;
- `site/src/components/seo-tools/types.ts`;
- `site/src/app/[slug]/page.tsx`;
- `Offer.documents`;
- `Offer.payoutMethods`;
- `Offer.approvalTone`.

Критичность: высокая для SEO-инструментов.

## 6. Калькулятор переплаты фильтрует офферы только по сумме

URL:

```text
http://localhost:3000/kalkulyator-pereplaty-po-zaymu#offers
```

Суть:

На показ офферов после расчета сейчас влияет только сумма займа. Срок и ставка меняют расчет переплаты, но не меняют список предложений.

Текущее сообщение:

```text
Показываем предложения, где максимальная сумма не ниже ...
```

Ожидаемое поведение:

Фильтр должен учитывать:

- сумму;
- срок;
- ставку.

Ожидаемая логика:

- сумма должна попадать в диапазон `minAmount/maxAmount`;
- срок должен попадать в диапазон `minTermDays/maxTermDays`, если он известен;
- ставка должна попадать в диапазон `dailyRateFrom/dailyRateTo`, если он известен;
- если у оффера поле индивидуальное/пустое, нужно явно решить правило:
  - либо показывать как `условия индивидуальны`;
  - либо исключать из точного фильтра;
  - либо показывать в отдельной секции `может подойти, условия уточняются`.

Обновить текст:

```text
Показываем предложения, которые подходят по сумме, сроку и указанной ставке, если эти условия известны.
```

Зона проекта:

- `site/src/components/seo-tools/overpayment-calculator.tsx`;
- `site/src/components/seo-tools/types.ts`;
- `site/src/app/[slug]/page.tsx`;
- `Offer.minAmount`;
- `Offer.maxAmount`;
- `Offer.minTermDays`;
- `Offer.maxTermDays`;
- `Offer.dailyRateFrom`;
- `Offer.dailyRateTo`.

Критичность: высокая для инструмента.

## 7. В админке оффера ключевые поля заполняются свободным текстом

URL:

```text
http://localhost:3000/admin?section=new-offer
```

Суть:

Поля, которые должны участвовать в фильтрации, заполняются вручную через textarea. Это делает фильтры ненадежными.

Проблемные поля:

- `Способы получения`;
- `Способы погашения`;
- `Требования`;
- `Документы`;
- `Плюсы/теги`;
- `Предупреждения`;
- частично `Время решения`;
- `Одобрение`.

Ожидаемое поведение:

Поля, влияющие на фильтрацию, должны заполняться через управляемые списки:

- чекбоксы;
- селекты;
- сегментированные переключатели;
- числовые поля;
- controlled tags из заранее заданного списка.

### Нормализовать способы получения

Минимальный список:

- `онлайн`;
- `на карту`;
- `СБП`;
- `наличные`;
- `банковский счет`.

Возможные будущие:

- `электронный кошелек`;
- `в офисе`;
- `курьером`.

### Нормализовать документы

Минимальный список:

- `паспорт`;
- `СНИЛС`;
- `ИНН`;
- `водительское удостоверение`;
- `справка о доходах`;
- `без обязательных документов` или отдельный boolean-флаг.

### Нормализовать способы погашения

Минимальный список:

- `банковская карта`;
- `СБП`;
- `наличные`;
- `банковский перевод`;
- `личный кабинет`;
- `терминал`.

### Нормализовать требования

Минимум:

- гражданство РФ;
- возраст от/до;
- регистрация;
- именная карта;
- подтверждение дохода;
- членство в КПК.

Важное решение:

Для фильтров нужны стабильные значения. Красивые публичные подписи можно маппить отдельно.

Зона проекта:

- `site/src/app/admin/offer-editor.tsx`;
- `site/src/app/admin/offer-actions.ts`;
- `site/prisma/schema.prisma`;
- возможно новый файл констант, например `site/src/lib/offer-options.ts`.

Критичность: высокая.

## 9. Нужна сортировка SEO-активов по типам и правильная логика вывода на главной

URL:

```text
http://localhost:3000/admin?section=seo&seoType=CATEGORY
```

Суть:

В админке SEO нужен функционал ручной сортировки элементов каждого типа выше/ниже по приоритету, как в управлении офферами.

Важное уточнение:

Лимит `4 места на главной` действует только для:

- статей;
- инструментов/сервисов.

Подборки лимитом 4 не ограничиваются.

### Общая логика сортировки

Когда выбран режим `Все типы`, ручной сортировки быть не должно.

Когда включен конкретный тип:

- `Подборки` → можно двигать подборки, порядок влияет на вывод на главной, но лимита 4 нет;
- `Статьи` → можно двигать статьи, топ-4 попадают на главную;
- `Сервисы` → можно двигать сервисы, топ-4 попадают на главную.

### Подборки

- На главной можно размещать сколько угодно подборок.
- Их будет не очень много.
- Позже через A/B-тесты и аналитику будем смотреть, с каких страниц сайта больше всего переходов к офферам, и оставлять наиболее эффективные.

### Статьи

- На главной показывать 4 приоритетные статьи.
- Остальные статьи выводить на отдельной странице блога.
- Страница блога должна показывать все статьи в хронологическом порядке по моменту размещения/публикации.

### Инструменты/сервисы

- На главной показывать 4 приоритетных инструмента.
- Остальные инструменты выводить на отдельной странице сервисов/инструментов.

### Верхнее меню

На главной в верхнем меню:

- `Сервисы` и `Статьи` могут оставаться якорями к 4 топовым блокам на главной;
- у каждого блока должна появиться кнопка:
  - `Все статьи` → страница блога;
  - `Все сервисы` / `Все инструменты` → страница инструментов.

Зона проекта:

- `site/src/app/admin/page.tsx`;
- `site/src/app/admin/seo-actions.ts`;
- модель `SeoPage`, возможно поле `displayPriority`;
- `site/src/app/page.tsx`;
- `site/src/components/site-header.tsx`;
- новые публичные страницы `/blog` и `/services` или `/tools`.

Критичность: средняя/высокая.

## 10. Подборки на главной нужно перенести сразу под первый экран

## Обновление 2026-07-09: новая обязательная проверка офферов

В проекте реализован MVP фильтра офферов по региону регистрации.
