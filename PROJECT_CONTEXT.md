## 1. Что это за проект

ZaimKarta — SEO-сайт-витрина микрозаймов на домене `zaimkarta.ru`.

Цель проекта:

- получать органический трафик из Яндекса;
- показывать пользователю полезное сравнение кредиторов/КПК/офферов;
- переводить пользователя по CPA-ссылке на сайт партнера или CPA-сети;
- фиксировать клики, `lead_id`, `click_id`, страницу и контекст перехода;
- развивать сайт не как тонкую прокладку, а как полезный финансовый сервис.

Главный принцип:

Сайт должен быть полезным и аккуратным: честные условия, понятные карточки, предупреждения о рисках, осторожные формулировки, удобные SEO-страницы и интерактивные инструменты.

## 2. Бизнес-модель и CPA

Планируемые CPA-сети на старте:

- Leads.su;
- Leadgid;
- прямые офферы или другие сети позже.

Основная логика:

1. Пользователь приходит из поиска на SEO-страницу.
2. Видит подборку, карточку, статью или интерактивный инструмент.
3. Нажимает CTA оффера.
4. Сайт ведет пользователя через внутренний маршрут `/go/[slug]`.
5. Сайт создает или читает cookie `zk_lead_id`.
6. Сайт создает `click_id`.
7. Сайт пишет событие `OfferClick`.
8. Сайт собирает партнерскую ссылку с параметрами:
   - `lead_id`;
   - `click_id`;
   - `sub1`;
   - `sub2`;
   - `sub3`.
9. Пользователь уходит на CPA/партнера.

`lead_id` — внутренний идентификатор визита/пользователя ZaimKarta, а не внешний ID CPA-сети. Один `lead_id` может иметь много кликов по разным офферам.

Ключевая реализация:

- `site/src/app/go/[slug]/route.ts`;
- `site/src/lib/cpa-click.ts`;
- модели `Lead`, `OfferClick`, `AffiliateOffer` в `site/prisma/schema.prisma`.

Публичная карточка должна быстро отвечать:

- кто выдает заем;
- сколько можно получить;
- на какой срок;
- ставка/ПСК;
- скорость рассмотрения;
- вероятность одобрения;
- способы получения;
- CTA для перехода.

Обязательные и важные поля:

- название;
- slug;
- статус;
- логотип или `logoText`;
- бейдж;
- рейтинг;
- количество отзывов;
- сумма;
- срок;
- ставка;
- ПСК;
- одобрение;
- время рассмотрения;
- способы получения;
- преимущества;
- предупреждения;
- юридическая сноска;
- дата проверки условий.

Что нельзя обещать как гарантию:

- `100% одобрение`;
- `без отказа`;
- `точно одобрят`;
- `деньги всем`;
- `без проверки`.

Безопаснее:

- `высокая вероятность одобрения`;
- `быстрое рассмотрение`;
- `для разных кредитных историй`;
- `решение принимает кредитор`.

Важное UI-решение:

Кнопка `Подробнее` в карточке оффера должна находиться на одной высоте внутри карточек независимо от количества строк выше. Для этого карточка сделана вертикальной flex-структурой, а CTA-блок прижат к нижней части карточки.

## 14. Фильтр офферов по региону регистрации

Идея:

Многие кредиторы и CPA-рекламодатели в микрозаймах имеют стоп-регионы на уровне рассмотрения заявки. Если пользователь с регистрацией в таком регионе видит оффер и кликает по нему, это ухудшает пользовательский опыт, снижает качество лида и тратит время пользователя.

Решение:

ZaimKarta должна учитывать регион регистрации пользователя и региональные ограничения офферов.

Продуктовая цель:

- показывать пользователю только те офферы, которые не ограничены для выбранного региона;
- повысить качество CPA-лида;
- сократить бесполезные переходы;
- улучшить доверие к витрине;
- превратить ZaimKarta в более умный предфильтр офферов.

Терминология:

- использовать `регион регистрации`;
- использовать `региональные ограничения`;
- избегать слов `бан`, `черный список`, `запрещенный регион`, `вам откажут`.

Админка оффера:

- добавить блок `Региональные ограничения`;
- использовать справочник регионов РФ в формате `код региона — название региона`, например `55 — Омская область`;
- дать админу выпадающий список с поисковой строкой;
- поиск должен работать по цифровому коду и названию региона;
- каждый найденный регион выбирается чекбоксом;
- выбранные регионы сохраняются в стоплист оффера;
- при редактировании оффера выбранные регионы должны восстанавливаться;
- данные предлагается хранить в `Offer.restrictedRegionCodes String[] @default([])`.

Публичный UX:

- при первом входе показать попап выбора региона регистрации;
- рекомендуемый текст: `Здравствуйте! Выберите регион вашей регистрации — так мы покажем предложения, которые с большей вероятностью подходят вам по условиям кредиторов.`;
- выбранный регион сохранить в cookie `zk_region_code`;
- дать возможность изменить регион позже;
- фильтровать офферы на главной, SEO-страницах, страницах офферов, SEO-инструментах и будущих витринах.

Серверная защита:

- `/go/[slug]` должен читать выбранный регион;
- если оффер ограничен для региона пользователя, не отправлять пользователя в CPA-сеть;
- вместо этого делать безопасный fallback redirect и позже показывать объяснение.

MVP реализован:

- справочник регионов РФ;
- поле `Offer.restrictedRegionCodes`;
- админский выбор регионов через поиск и чекбоксы;
- попап выбора региона;
- cookie `zk_region_code`;
- фильтрация публичных офферов;
- проверка `/go/[slug]`;
- документация в `docs/region-registration-offer-filter.md`.

Ключевые файлы реализации:

- `site/src/lib/russian-regions.ts` — справочник регионов и нормализация кодов;
- `site/src/lib/region-cookie.ts` — чтение cookie выбранного региона на сервере;
- `site/src/components/region-registration-control.tsx` — публичный выбор региона;
- `site/src/app/admin/region-restrictions-field.tsx` — админский выбор стоп-регионов оффера;
- `site/src/lib/offers.ts` — фильтрация активных офферов по региону;
- `site/src/lib/cpa-click.ts` — серверная защита `/go/[slug]`;

- сгенерировать длинные уникальные значения;
- хранить их в менеджере паролей;
- не класть реальные секреты в GitHub;
- менять вручную при подозрении на утечку, смене подрядчика, взломе или переезде.

2FA для админки:

- полезна и желательна;
- не первый шаг;
- сначала закрывается базовая защита;
- затем отдельным этапом можно добавить TOTP через Google Authenticator, Яндекс Ключ, 1Password или Bitwarden.

Принятый порядок security-работ:

1. Production env и секреты.
2. Базовая защита админки.
3. Защита `/go/[slug]`.
4. 2FA для админки.
5. Загрузка логотипов.
6. VPS-защита.
7. Бэкапы.
8. Мониторинг и аварийная реакция.

Следующий рекомендуемый отдельный рабочий диалог:

```text
Этап 1: production env и секреты ZaimKarta
```

Задачи для него:

- подготовить безопасный `production.env.example`;
- проверить `.gitignore`;
- описать владельцу, какие значения сгенерировать;
- не записывать реальные секреты в репозиторий;
- проверить сборку с тестовыми production-переменными.

Проверки, которые уже использовались:

```bash
cd site
npm run lint
NEXT_PUBLIC_SITE_URL=https://zaimkarta.ru ADMIN_SESSION_SECRET=local-build-admin-secret-1234567890abcdef LEAD_IP_HASH_SALT=local-build-ip-hash-salt-1234567890abcdef npm run build
```

Локально можно проверять кодовую часть безопасности: секреты, вход в админку, rate limit, `/robots.txt`, headers/noindex, `/go/[slug]`, загрузку логотипов, Prisma/PostgreSQL, миграции и production build.

Только на VPS полноценно проверяются HTTPS, firewall, fail2ban, закрытие портов, DNS, DDoS-защита провайдера, внешние uptime-алерты и бэкапы вне сервера.

## 17. Как обновлять этот файл

Этот файл нужно обновлять после каждого смыслового решения:

- меняется бизнес-логика;
- добавляется новая сущность;
- меняется CPA-трекинг;
- меняется SEO-архитектура;
- меняются правила оффера;
- появляется комплаенс-вывод;
- меняется процесс разработки;
- появляется новый риск или ограничение.

Формула обновления:

```text
Что решили?
Почему?
Где реализовано?
Какие файлы связаны?
Какие риски или ограничения появились?
```

Не нужно записывать сюда каждую мелкую правку CSS. Нужно записывать то, что помогает восстановить мышление проекта.
