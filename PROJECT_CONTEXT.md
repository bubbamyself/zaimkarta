## 1. Что это за проект

ZaimKarta — SEO-сайт-витрина микрозаймов на домене `zaimkarta.ru`.

Цель проекта:

- получать органический трафик из Яндекса;
- показывать пользователю полезное сравнение МФО/КПК/офферов;
- переводить пользователя по CPA-ссылке на сайт МФО или CPA-сети;
- фиксировать клики, `lead_id`, `click_id`, страницу и контекст перехода;
- развивать сайт не как тонкую прокладку, а как полезный финансовый сервис.

Главный принцип:

Сайт должен быть полезным и аккуратным: честные условия, понятные карточки, предупреждения о рисках, осторожные формулировки, удобные SEO-страницы и интерактивные инструменты.

## 2. Бизнес-модель и CPA

Планируемые CPA-сети на старте:

- Leads.su;
- Leadgid;
- прямые офферы или другие сети позже.

Основная логика:

1. Пользователь приходит из поиска на SEO-страницу.
2. Видит подборку, карточку, статью или интерактивный инструмент.
3. Нажимает CTA оффера.
4. Сайт ведет пользователя через внутренний маршрут `/go/[slug]`.
5. Сайт создает или читает cookie `zk_lead_id`.
6. Сайт создает `click_id`.
7. Сайт пишет событие `OfferClick`.
8. Сайт собирает партнерскую ссылку с параметрами:
   - `lead_id`;
   - `click_id`;
   - `sub1`;
   - `sub2`;
   - `sub3`.
9. Пользователь уходит на CPA/МФО.

`lead_id` — внутренний идентификатор визита/пользователя ZaimKarta, а не внешний ID CPA-сети. Один `lead_id` может иметь много кликов по разным офферам.

Ключевая реализация:

- `site/src/app/go/[slug]/route.ts`;
- `site/src/lib/cpa-click.ts`;
- модели `Lead`, `OfferClick`, `AffiliateOffer` в `site/prisma/schema.prisma`.

## 6. Офферы и карточки

Главный документ по модели оффера:

```text
offer-card-model.md
```

Публичная карточка должна быстро отвечать:

- кто выдает заем;
- сколько можно получить;
- на какой срок;
- ставка/ПСК;
- скорость рассмотрения;
- вероятность одобрения;
- способы получения;
- CTA для перехода.

Обязательные и важные поля:

- название;
- slug;
- статус;
- логотип или `logoText`;
- бейдж;
- рейтинг;
- количество отзывов;
- сумма;
- срок;
- ставка;
- ПСК;
- одобрение;
- время рассмотрения;
- способы получения;
- преимущества;
- предупреждения;
- юридическая сноска;
- дата проверки условий.

Что нельзя обещать как гарантию:

- `100% одобрение`;
- `без отказа`;
- `точно одобрят`;
- `деньги всем`;
- `без проверки`.

Безопаснее:

- `высокая вероятность одобрения`;
- `быстрое рассмотрение`;
- `для разных кредитных историй`;
- `решение принимает МФО`.

Важное UI-решение:

Кнопка `Подробнее` в карточке оффера должна находиться на одной высоте внутри карточек независимо от количества строк выше. Для этого карточка сделана вертикальной flex-структурой, а CTA-блок прижат к нижней части карточки.

Файл:

```text

```

## 8. SEO-страницы

Главный документ по SEO-системе:

```text
docs/seo-tools-strategy.md
```

SEO-страницы лежат в модели `SeoPage`.

Типы страниц:

- `CATEGORY` — коммерческая подборка;
- `ARTICLE` — информационная статья;
- `SERVICE` — страница интерактивного инструмента.

Статусы:

- `DRAFT`;
- `PUBLISHED`;
- `PAUSED`;
- `ARCHIVED`.

Intent:

- `COMMERCIAL`;
- `INFORMATIONAL`;
- `SERVICE`;
- `MIXED`.

Связанные модели:

- `SeoPageOffer` — офферы на странице, порядок и контекст;
- `SeoPageFaqItem` — FAQ;
- `SeoPageTool` — привязанные интерактивные инструменты.

Принцип:

SEO-страница должна закрывать конкретный интент, а не быть универсальным полотном. Коммерческие страницы должны держать офферы близко к началу, статьи не должны выглядеть как замаскированные витрины, сервисные страницы должны выводить инструмент близко к началу.

## 12. Комплаенс

Комплаенс для проекта — это база правил, которым сайт должен соответствовать:

- финансовая реклама;
- раскрытие условий;
- осторожные формулировки;
- CPA-ограничения;
- пользовательские данные;
- дисклеймеры;
- правила карточек;
- правила SEO-контента.

Для займов нельзя строить комплаенс только из головы. Нужен research по:

- официальным источникам Банка России;
- ФАС и рекламе финансовых услуг;
- 353-ФЗ о потребительском кредите/займе;
- правилам CPA-сетей;
- крупным конкурентам и их паттернам раскрытия условий.

Важные принципы уже приняты:

- не обещать гарантированное одобрение;
- явно писать, что решение принимает МФО;
- показывать ПСК/ставку/срок/сумму максимально прозрачно;
- отделять публичные данные от CPA-ограничений;
- хранить дату проверки условий;
- не собирать лишние персональные данные в SEO-инструментах.

Нужно создать отдельную папку:

```text
docs/compliance/
```

Рекомендуемые файлы:

```text
docs/compliance/README.md
docs/compliance/source-map.md
docs/compliance/content-rules.md
docs/compliance/offer-card-rules.md
docs/compliance/advertising-rules.md
docs/compliance/privacy-rules.md
docs/compliance/cpa-rules.md
docs/compliance/decision-log.md
```

Google Doc может быть человекочитаемой версией, но source of truth для разработки лучше держать в GitHub.

## 13. Пользовательские данные и приватность

Сейчас проект не должен превращаться в сервис сбора заявок.

Что уже есть:

- cookie `zk_lead_id`;
- `userAgent`;
- `referrer`;
- UTM;
- hash IP при наличии `x-forwarded-for`;
- история кликов по офферам.

Важно:

- не хранить лишние персональные данные;
- для интерактивных инструментов избегать ФИО, телефона, паспорта, email;
- результаты калькуляторов и чеклистов должны быть приблизительными и объясненными;
- нужен будущий документ `privacy-rules.md`.
