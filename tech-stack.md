# Технический стек проекта

## Короткая рекомендация

Для первой версии проекта я рекомендую такой стек:

- сайт: Next.js;
- язык: TypeScript;
- база данных: PostgreSQL;
- работа с базой: Prisma;
- админка: простая встроенная админка на старте;
- стили: Tailwind CSS;
- аналитика: Яндекс Метрика плюс собственный учет кликов;
- хостинг первой версии: VPS с Docker или управляемая платформа;
- домен: `zaimkarta.ru`;
- CDN и защита: позже, после запуска и первых данных.

Это не самый простой стек в мире, но он хороший для сайта, который должен расти, индексироваться, хранить данные и зарабатывать на партнерских переходах.

## Почему не Tilda

Tilda хороша для лендингов и быстрых тестов, но для нашей задачи она слабая:

- сложно строить много SEO-страниц по шаблонам;
- неудобно управлять базой МФО;
- ограниченный контроль над скоростью и HTML;
- неудобный собственный трекинг кликов;
- сложно делать нормальные фильтры, карточки, подборки и внутреннюю перелинковку;
- проект быстро упрется в потолок.

Нам нужен не лендинг, а управляемый SEO-сайт с данными.

## Варианты движка

### Вариант 1. WordPress

Плюсы:

- много готовых плагинов;
- легко найти исполнителей;
- есть админка из коробки;
- можно быстро публиковать статьи.

Минусы:

- часто получается тяжелым и медленным;
- много зависимости от плагинов;
- сложнее сделать аккуратный трекинг партнерских кликов;
- выше риск технического мусора;
- при росте проекта нужно постоянно следить за безопасностью.

Вывод: можно, но я бы не выбирал WordPress для этого проекта, если мы хотим сделать аккуратный самописный продукт.

### Вариант 2. Astro

Плюсы:

- очень быстрый для контентных сайтов;
- хорошо подходит для статей и SEO-страниц;
- проще, чем Next.js, если сайт почти полностью статический.

Минусы:

- когда появляется админка, база, трекинг, динамические фильтры и партнерские переходы, архитектура становится менее очевидной;
- меньше готовых паттернов для полноценного приложения;
- может быть отлично, но нужно аккуратнее проектировать backend-часть.

Вывод: хороший вариант для чистого контентного сайта, но для витрины с базой и трекингом я бы выбрал не его.

### Вариант 3. Next.js

Плюсы:

- хорошо подходит и для SEO-страниц, и для динамических функций;
- можно отдавать поисковикам готовый HTML;
- удобно делать каталог, карточки МФО, статьи, подборки и API;
- можно построить встроенную админку;
- хорошо поддерживает самохостинг через Node.js или Docker;
- много специалистов и материалов.

Минусы:

- сложнее, чем Tilda или простой статический сайт;
- нужно аккуратно настроить сборку, кэширование и деплой;
- без дисциплины проект можно усложнить.

Вывод: лучший баланс для нашего проекта.

## Почему Next.js подходит для SEO

Для SEO нам важно, чтобы поисковик сразу видел нормальную HTML-страницу: заголовок, текст, карточки, ссылки, мета-теги. Next.js позволяет делать такие страницы, а не отдавать пустую оболочку, которая потом дорисовывается в браузере.

Это важно для:

- страниц подборок;
- карточек МФО;
- статей;
- главной страницы;
- sitemap.xml;
- внутренней перелинковки.

## База данных

Рекомендация: PostgreSQL.

Почему:

- надежная и распространенная база;
- хорошо подходит для структурированных данных;
- удобно хранить МФО, статьи, клики, партнерские ссылки;
- легко переносить между хостингами;
- не привязывает нас к одному сервису.

На старте в базе будут таблицы:

- МФО;
- партнерские ссылки;
- клики;
- SEO-страницы;
- статьи;
- категории;
- пользователи админки.

Локально база запускается через Docker Desktop. Для этого в корне проекта создан файл `docker-compose.yml` с контейнером PostgreSQL.

Локальные параметры:

- база: `zaimkarta`;
- пользователь: `zaimkarta`;
- порт: `5432`;
- строка подключения хранится в `site/.env`.

## Prisma

Prisma — это прослойка между кодом сайта и базой данных.

По-человечески: вместо того чтобы вручную писать много сложных запросов к базе, мы описываем структуру данных, а Prisma помогает безопасно читать и записывать данные.

Зачем нам Prisma:

- меньше ошибок в работе с базой;
- понятная схема данных;
- проще развивать проект;
- удобно делать миграции, то есть изменения структуры базы.

## Админка

На старте я рекомендую простую встроенную админку.

В ней должны быть разделы:

- МФО;
- партнерские ссылки;
- статьи;
- SEO-подборки;
- клики и статистика;
- настройки сайта.

Почему не брать большую CMS сразу:

- она добавит сложность;
- часть функций нам не понадобится;
- сначала важнее понять реальную структуру данных;
- встроенную админку можно сделать ровно под нашу задачу.

Позже, если проект вырастет, можно будет перейти на более мощное CMS-решение.

## Трекинг кликов

Это одна из самых важных частей проекта.

На старте проект планируется подключать к CPA-сетям Leads.su и Leadgid. Поэтому трекинг должен поддерживать разные форматы партнерских ссылок и разные наборы параметров для разных офферов.

Как будет работать переход:

1. Пользователь нажимает кнопку МФО.
2. Сайт ведет его не сразу на МФО, а на наш технический адрес, например:

`/go/zaymer`

3. Сайт проверяет cookie текущего визита.
4. Если `lead_id` еще нет, сайт создает его.
5. Сайт записывает клик по офферу в базу как событие внутри этого `lead_id`.
6. Сайт собирает партнерскую ссылку с параметрами.
7. Пользователь автоматически перенаправляется на сайт МФО или CPA-сеть.

Пример итоговой ссылки:

`https://partner.example.com/?webmaster_id=123&offer_id=456&sub1=zaimy-na-kartu&sub2=lead_abc123`

Что мы сможем анализировать:

- сколько офферов прокликал один пользователь в рамках визита;
- с какой страницы пришел переход;
- какая МФО получает больше кликов;
- какие SEO-страницы потенциально зарабатывают;
- какие кнопки и блоки работают лучше;
- сколько кликов уходит в каждую партнерку.

## Яндекс Метрика

Яндекс Метрика нужна обязательно.

Что настраиваем:

- счетчик;
- цели на клики по МФО;
- цели на использование фильтров;
- Вебвизор;
- карту кликов;
- события для важных действий.

Метрика покажет поведение людей на сайте, а наша база покажет точные партнерские клики.

## Хостинг

Тут есть два разумных пути.

### Путь 1. Управляемая платформа

Примеры: Vercel, Render, Railway и похожие сервисы.

Плюсы:

- проще запуск;
- меньше серверной рутины;
- удобно для разработки.

Минусы:

- может быть дороже при росте;
- часть сервисов зарубежные;
- возможны нюансы с оплатой, доступностью и юрисдикцией;
- база часто живет отдельно.

### Путь 2. VPS

Примеры: Timeweb Cloud, Selectel, Yandex Cloud, другие VPS-провайдеры.

Плюсы:

- больше контроля;
- можно разместить сайт и базу рядом;
- понятнее расходы;
- лучше подходит для долгосрочного независимого проекта;
- можно выбрать сервер ближе к основной аудитории.

Минусы:

- нужно настроить сервер;
- нужно следить за обновлениями, резервными копиями и безопасностью;
- чуть больше ответственности.

Предварительная рекомендация: для первой рабочей версии можно идти через VPS, но настроить все максимально просто: Docker, автоматический HTTPS, резервные копии базы.

Домен уже куплен через Beget: `zaimkarta.ru`. Beget можно рассмотреть и как площадку для размещения, но решение по хостингу лучше принять после проверки, насколько удобно там запускать выбранный стек: Next.js, PostgreSQL и деплой через Docker или похожий механизм.

## Что такое Docker и зачем он нам

Docker — это способ упаковать сайт со всеми его зависимостями.

Проще говоря: мы не вручную настраиваем сервер “по памяти”, а описываем, как сайт должен запускаться. Это снижает вероятность хаоса при переносе или обновлении.

На сервере будут:

- приложение Next.js;
- база PostgreSQL;
- служебные настройки;
- механизм обновления сайта.

## Почта

На старте сложная почтовая система не нужна.

Достаточно:

- отдельной почты на домене;
- формы обратной связи, если она действительно нужна;
- технической почты для уведомлений админа.

## Безопасность

Минимум на старте:

- HTTPS;
- закрытая админка;
- сложный пароль;
- ограничение попыток входа;
- резервные копии базы;
- отсутствие хранения паспортных данных и анкет пользователей;
- аккуратная политика конфиденциальности.

Важно: на первом этапе лучше не собирать персональные данные заемщиков. Мы не МФО и не должны превращаться в сервис приема заявок, пока для этого нет юридической и технической готовности.

## Обновление 2026-07-09: региональный фильтр офферов

В текущей кодовой базе реализован MVP фильтра офферов по региону регистрации.

Технически это добавляет к стеку несколько важных элементов:

- Prisma-поле `Offer.restrictedRegionCodes String[] @default([])`;
- миграцию `20260630090000_offer_restricted_region_codes`;
- справочник регионов РФ в `site/src/lib/russian-regions.ts`;
- cookie `zk_region_code` для выбранного региона пользователя;
- серверную проверку CPA-перехода в `site/src/lib/cpa-click.ts`;
- публичный компонент выбора региона в `site/src/components/region-registration-control.tsx`;
- админский компонент региональных ограничений в `site/src/app/admin/region-restrictions-field.tsx`.

Проверки после внедрения:

```bash
cd site
npm run db:generate
npm run db:migrate
npm run lint
npx tsc --noEmit
npm run build
```

Эта фича не добавляет новых библиотек и остается внутри текущего стека `Next.js + TypeScript + PostgreSQL + Prisma + Tailwind CSS`.
