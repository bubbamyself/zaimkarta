# ZaimKarta — отчёт по информационной безопасности: этап 6А

**Дата:** 13 июля 2026 года  
**Этап:** 6А — локальная подготовка production-инфраструктуры VPS  
**Статус:** локальная подготовка завершена; публичный production-запуск пока запрещён

## 1. Резюме для ответственного за ИБ

Для ZaimKarta подготовлен минимальный production-контур будущего VPS:

```text
Internet → Caddy → Next.js → PostgreSQL
```

Архитектура разделяет публичный вход, приложение и базу данных. Из интернета должны быть доступны только порты reverse proxy `80` и `443`, а также отдельно ограниченный административный SSH-порт. Порты приложения `3000` и PostgreSQL `5432` в production Compose наружу не публикуются.

Текущая работа выполнена только локально. Реальный VPS, DNS, firewall, SSH, fail2ban, production-секреты, внешние бэкапы и мониторинг ещё не настроены. Поэтому этап 6А подтверждает готовность шаблонов, но не подтверждает защищённость реального сервера.

## 2. Что подготовлено

| Область | Реализованное решение | Статус |
| --- | --- | --- |
| Контейнерный контур | Отдельный production Compose для Caddy, Next.js и PostgreSQL | Готов локальный пример |
| Сетевая изоляция БД | PostgreSQL находится в закрытой Docker-сети, публикации `5432` нет | Готово в примере |
| Изоляция приложения | Next.js слушает `3000` только внутри Docker, публикации порта нет | Готово в примере |
| Публичный вход | Только Caddy публикует `80/443` | Готово в примере |
| HTTPS | Caddy должен автоматически получать и обновлять сертификат | Требует проверки на VPS |
| Домены | `www.zaimkarta.ru` перенаправляется на `zaimkarta.ru` | Готово в примере |
| Клиентский IP | Proxy перезаписывает `X-Forwarded-For` и `X-Real-IP`, снижая риск подмены посетителем | Готово в примере, требует проверки на VPS |
| Размер запроса | Caddy ограничивает тело запроса до 1 МБ; приложение отдельно ограничивает SVG-логотип до 400 КБ | Готово в конфигурации |
| Загружаемые файлы | Для `public/uploads/logos` предусмотрен постоянный Docker-volume | Готово; внешний бэкап не настроен |

| Миграции | При старте приложение выполняет `prisma migrate deploy` после готовности БД | Готово в Dockerfile |
| Логи proxy | Access-логи Caddy направляются в стандартный вывод Docker в JSON | Готово; ротация не настроена |

## 3. Модель доступа и открытые порты

Планируемая политика внешнего доступа:

- `80/tcp` — HTTP-редирект и получение HTTPS-сертификата;
- `443/tcp` — основной HTTPS-трафик;
- `443/udp` — HTTP/3;
- SSH-порт — административный доступ только по ключу и с ограничениями, которые настраиваются на этапе 6Б;
- `3000/tcp` — закрыт от интернета;
- `5432/tcp` — закрыт от интернета;
- остальные порты — закрыты по умолчанию.

Важно: отсутствие публикации портов в Docker Compose является одним уровнем защиты, но не заменяет firewall. Фактическую недоступность `3000` и `5432` необходимо проверить с внешнего узла после создания VPS.

## 4. Работа с секретами

В репозитории находится только `deploy/production.env.server.example` с заведомо нерабочими заглушками. Настоящий файл `deploy/production.env.server` добавлен в `.gitignore`.

Перед запуском должны быть независимо сгенерированы:

- `POSTGRES_PASSWORD`;
- `ADMIN_SESSION_SECRET`;
- `LEAD_IP_HASH_SALT`;
- пароль production-администратора.

Секреты должны храниться в менеджере паролей и в закрытом env-файле на VPS с правами только для владельца. Передача секретов через Git, задачи, чаты, письма и скриншоты запрещена.

## 5. Выполненные проверки

- production Compose успешно прошёл `docker compose config --quiet`;
- dev-файл `docker-compose.yml` не изменён;
- `npm run lint` завершился без ошибок; присутствуют три ранее существовавших предупреждения о применении HTML-тега `<img>`;
- `git diff --check` не выявил ошибок форматирования diff;
- поиск характерных признаков приватных ключей и токенов в новых файлах ничего не выявил;
- подтверждено отсутствие публикации `3000` и `5432` в production Compose;
- параметры reverse proxy сверены с официальной документацией Caddy.

Полная сборка и запуск контейнеров не выполнялись, поскольку локально отсутствовали необходимые Docker-образы Node и Caddy. Это не позволяет считать runtime-проверку завершённой.

## 6. Известные риски и ограничения

### Критичные до публичного запуска

1. Реальный VPS ещё не защищён: не настроены SSH по ключу, запрет root/password login, firewall и fail2ban.
2. Не проверены DNS, автоматический HTTPS и редиректы на настоящем домене.
3. Не создан и не проверен production env с реальными уникальными секретами.
4. Не выполнена внешняя проверка закрытия портов `3000` и `5432`.
5. Не настроены внешние ежедневные бэкапы PostgreSQL и загруженных логотипов.
6. Не проверено восстановление из бэкапа.
7. Не настроены uptime-, disk-, CPU/RAM- и error-уведомления.
8. Не выполнен production smoke test приложения, админки, загрузок и CPA-переходов.

### Отдельный риск `/go/`

В приложении уже действует in-memory rate limit, однако стандартная сборка Caddy не предоставляет rate limit без стороннего модуля. До публичного запуска необходимо выбрать и проверить дополнительное ограничение `/go/` на уровне провайдера/WAF либо отдельно согласовать Nginx или custom-сборку Caddy.

До закрытия этого пункта защиту CPA-переходов нельзя считать полностью принятой для production.

### Эксплуатационные ограничения MVP

- Next.js в текущей архитектуре обращается к PostgreSQL во время production-сборки, поэтому контейнер выполняет сборку после готовности БД. Это увеличивает время первого запуска и перезапуска.
- Docker-volume защищает логотипы от потери при пересоздании контейнера, но не защищает от потери VPS или диска.
- Логи направляются в Docker, но политика ротации и срок хранения ещё не заданы.
- Конфигурация рассчитана на один экземпляр приложения; при горизонтальном масштабировании in-memory rate limit потребует общего хранилища или внешнего ограничения.

## 7. Условия допуска к публичному запуску

Ответственному за ИБ рекомендуется не согласовывать публичный запуск, пока не будут подтверждены следующие пункты:

- VPS обновлён и настроен отдельный deploy-пользователь;
- SSH работает только по ключу, вход root и по паролю запрещён;
- firewall и fail2ban включены;
- извне доступны только согласованные `80`, `443` и SSH-порт;
- `3000` и `5432` недоступны с внешнего узла;
- HTTPS и доменные редиректы проверены;
- production-секреты уникальны, сохранены безопасно и отсутствуют в Git;
- миграции, сборка, healthcheck и smoke test проходят;
- proxy/provider rate limit для `/go/` выбран и проверен;
- внешние бэкапы базы и логотипов выполняются автоматически;
- тестовое восстановление успешно выполнено;
- настроены мониторинг и аварийные уведомления;
- из production-базы удалены или отключены тестовые администраторы и demo CPA-ссылки.

## 8. Итоговый вывод

Этап 6А выполнен в объёме локальной подготовки: создан понятный и минимальный инфраструктурный комплект, база и приложение изолированы от прямой публикации, предусмотрены HTTPS-proxy, постоянное хранение логотипов и безопасный шаблон env.

**ИБ-статус: условно готово к переходу на этап 6Б, не готово к публичной эксплуатации.**

Следующая контрольная точка — развёртывание на реальном VPS и документальное подтверждение каждого пункта раздела «Условия допуска к публичному запуску».
