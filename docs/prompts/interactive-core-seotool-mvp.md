# Prompt: Interactive Core / SeoTool MVP

```text
Ты работаешь как product architect, UX designer и senior full-stack engineer для проекта ZaimKarta.

Контекст:
ZaimKarta — это SEO-витрина микрозаймов. Владелец сам добавляет офферы, сам выбирает CPA-ссылки и сам управляет тем, какие офферы показываются на сайте.

Проект НЕ является:
- маркетплейсом для CPA-сетей;
- кабинетом рекламодателей;
- форумом;
- сервисом пользовательских отзывов;
- личным кабинетом заемщика;
- сложной CMS уровня WordPress.

Что уже реализовано:
1. Админка офферов:
   - создание/редактирование офферов;
   - публичные данные;
   - партнерская ссылка;
   - статус;
   - приоритет;
   - дата проверки условий.

2. CPA-переход:
   - `/go/[slug]`;
   - `lead_id`;
   - `click_id`;
   - `OfferClick`;
   - tracking/sub-параметры;
   - редирект на CPA-ссылку.

3. SEO-страницы:
   - `SeoPage`;
   - `SeoPageOffer`;
   - `SeoPageFaqItem`;
   - типы страниц: `CATEGORY`, `ARTICLE`, `SERVICE`;
   - статусы: `DRAFT`, `PUBLISHED`, `PAUSED`, `ARCHIVED`;
   - публичная `/[slug]`;
   - админка SEO-страниц;
   - выбор офферов;
   - порядок офферов;
   - FAQ.

Главное продуктовое решение:
ZaimKarta нужно развивать как SEO-систему с reusable-интерактивными инструментами.

Важно:
Сервисы не откладываются “на потом”. Интерактивные инструменты — это ядро будущей SEO-системы.

Нужно разделить:
- `SeoPage` — публичная SEO-страница;
- `SeoTool` — переиспользуемый интерактивный инструмент;
- `SeoPageTool` — подключение инструмента к странице;
- `contentBlocks` — простая структура тела страницы;
- `SeoPageOffer` — контекстная связь страницы с оффером.

Инструмент должен уметь:
- иметь самостоятельную сервисную страницу;
- встраиваться в подборку;
- встраиваться в статью;
- позже использоваться на главной.

Примеры:
- калькулятор переплаты внутри подборки “Займы на карту”;
- чек-лист готовности к заявке внутри статьи;
- мини-подборщик офферов внутри коммерческой страницы;
- отдельная страница “Калькулятор переплаты по займу”.

Текущая задача:
Спроектировать первый практический этап реализации интерактивного ядра ZaimKarta.

Нужно получить не абстрактное исследование, а конкретное ТЗ для разработки MVP.

Исследуй и спроектируй:

1. Модель `SeoTool`
Предложи финальную MVP-структуру:
- id;
- slug;
- type;
- status;
- name;
- title;
- description;
- config;
- defaultBlock;
- createdAt;
- updatedAt.

Типы инструментов:
- OVERPAYMENT_CALCULATOR;
- APPLICATION_CHECKLIST;
- MINI_OFFER_PICKER;
- LOAN_TYPE_QUIZ;
- COMPARISON.

Но для MVP нужно реализовать только:
- калькулятор переплаты;
- чек-лист готовности к заявке.

2. Модель `SeoPageTool`
Спроектируй связь страницы с инструментом:
- pageId;
- toolId;
- position;
- blockId;
- variant;
- title;
- intro;
- config.

Объясни, что хранится в самом `SeoTool`, а что можно переопределить на конкретной странице через `SeoPageTool`.

3. `contentBlocks`
Предложи MVP-структуру JSON-блоков:
- paragraph;
- heading;
- list;
- callout;
- offers;
- tool;
- faq;
- riskNotice;
- cta.

Важно:
Не строить visual builder. Нужна простая структура, которую реально поддерживать одному владельцу проекта.

Критически важно:
Не предлагай сырой JSON как основной интерфейс админки. JSON может быть внутренним форматом хранения для `SeoTool.config` и `SeoPage.contentBlocks`, но владелец сайта должен работать через понятные формы:
- input;
- textarea;
- select;
- checkbox/toggle;
- repeatable lists;
- кнопки добавления, удаления и перемещения блоков.

Для любой JSON-backed части нужно заложить mapper:
- form fields -> structured JSON;
- structured JSON -> form fields.

Raw JSON допустим только как developer/debug view, но не как основной рабочий интерфейс.

4. Первый инструмент: калькулятор переплаты
Спроектируй:
- пользовательский сценарий;
- какие поля вводит пользователь;
- какие значения по умолчанию;
- как считается результат;
- какие ограничения ввода;
- какой текст результата показывается;
- где CTA;
- какие офферы показывать после результата;
- какой risk notice нужен;
- как этот калькулятор выглядит при встраивании в подборку;
- как выглядит на самостоятельной сервисной странице.

Важно:
Калькулятор не должен обещать точный расчет договора. Он показывает ориентировочную переплату.

5. Второй инструмент: чек-лист готовности к заявке
Спроектируй:
- пользовательский сценарий;
- вопросы;
- варианты ответов;
- как считается результат;
- какие советы показываются;
- где CTA;
- какие офферы показываются после результата;
- как чек-лист встраивается в статью;
- как выглядит на самостоятельной сервисной странице.

6. Админка инструментов
Спроектируй раздел “Интерактивные инструменты”:
- список инструментов;
- создание инструмента;
- редактирование инструмента;
- статус;
- тип;
- базовый config;
- предпросмотр;
- где используется;
- проверка перед публикацией.

Важно:
Инструмент создается отдельно. В подборках и статьях он только подключается.

Для калькулятора и чек-листа нужно проектировать type-specific формы, а не textarea с JSON.

7. Подключение инструмента к SEO-странице
Спроектируй UX в редакторе страницы:
- блок “Интерактивные сервисы”;
- выбрать существующий инструмент;
- выбрать вариант отображения: full / compact / inline;
- задать локальный заголовок;
- задать intro;
- выбрать позицию;
- удалить подключение со страницы без удаления инструмента.

8. Публичный renderer
Опиши, как должен работать renderer:
- если блок `tool`, найти `SeoTool`;
- проверить статус инструмента;
- применить локальные настройки `SeoPageTool`;
- отрендерить нужный React-компонент;
- если инструмент неактивен, показать fallback или скрыть блок.

9. Проверки публикации
Предложи проверки:
Для `SeoTool`:
- указан тип;
- указан name/title;
- config валиден;
- есть публичный компонент;
- активный инструмент нельзя сломать, если он используется на опубликованных страницах.

Для `SeoPage`:
- если подключен инструмент, он должен быть ACTIVE;
- не должно быть битых ссылок на инструменты;
- сервисная страница должна иметь основной инструмент;
- статья/подборка могут не иметь инструмента.

10. Минимальные изменения БД
Предложи Prisma migration:
- enums;
- `SeoTool`;
- `SeoPageTool`;
- `SeoPage.contentBlocks`;
- возможно `SeoPage.intent`;
- возможно расширение `SeoPageOffer`.

Но не раздувай MVP. Если что-то можно отложить — отложи.

11. Очередность разработки
Дай практичный порядок:
- миграция;
- seed демо-инструментов;
- backend actions;
- админка инструментов;
- публичные компоненты калькулятора и чек-листа;
- подключение к SEO-страницам;
- renderer;
- проверки;
- тестирование.

12. Что не делать сейчас
Не предлагать:
- сложный comparison tool;
- личные кабинеты;
- сбор персональных данных;
- регистрацию;
- сохранение результатов пользователя;
- сложную аналитику;
- A/B-тесты;
- CPA API;
- полноценный visual builder;
- сложную систему рейтинга;
- форум;
- отзывы.

Формат результата:
1. Краткое резюме решения.
2. Финальная MVP-модель данных.
3. Структура `SeoTool.config` для калькулятора.
4. Структура `SeoTool.config` для чек-листа.
5. UX публичного калькулятора.
6. UX публичного чек-листа.
7. UX админки инструментов.
8. UX подключения инструмента к странице.
9. Renderer logic.
10. Проверки публикации.
11. Пошаговый development backlog.
12. Риски и что отложить.

Пиши практично, как PM/CTO и senior engineer. Цель — подготовить четкое ТЗ для реализации интерактивного ядра ZaimKarta: `SeoTool`, `SeoPageTool`, первый калькулятор и первый чек-лист.
```

## Обновление контекста 2026-07-09

В проекте уже реализован MVP фильтра офферов по региону регистрации.

Если этот промпт используется в новом диалоге, нужно учитывать:

- пользовательский регион хранится в cookie `zk_region_code`;
- ограничения оффера хранятся в `Offer.restrictedRegionCodes`;
- любые будущие SEO-инструменты, которые показывают офферы, должны применять региональную фильтрацию;
- `/go/[slug]` дополнительно защищает CPA-переход на сервере.
