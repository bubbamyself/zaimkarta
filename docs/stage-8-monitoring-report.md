# ZaimKarta — отчёт по этапу 8: production-мониторинг

Дата закрытия: 14 июля 2026 года.

Статус: выполнено.

## Итог

Этап 8 закрыт: мониторинг и аварийные уведомления работают, владелец получает алерты.

Для MVP используется бесплатная и проверяемая схема без Prometheus, Grafana, Loki, Sentry и платного heartbeat:

- UptimeRobot проверяет сайт снаружи;
- локальный health-check на VPS проверяет состояние сервера, Docker, сайта и бэкапов;
- общий результат локальных проверок публикуется через безопасный endpoint `/api/health/server`;
- UptimeRobot отправляет владельцу email при переходе монитора в Down и после восстановления.

## Что настроено

В UptimeRobot работают пять HTTP(s)-мониторов с интервалом 5 минут:

| Адрес | Назначение |
| --- | --- |
| `https://zaimkarta.ru/` | Доступность главной страницы и HTTPS |
| `https://zaimkarta.ru/robots.txt` | Доступность SEO-инфраструктуры |
| `https://zaimkarta.ru/admin/login` | Доступность страницы входа в админку |
| `https://zaimkarta.ru/sitemap.xml` | Доступность sitemap |
| `https://zaimkarta.ru/api/health/server` | Общий результат серверных проверок |

Реальный `/go/[slug]` намеренно не добавлен во внешний мониторинг: автоматические запросы могли бы создавать ложные CPA-клики.

На VPS установлен `/usr/local/sbin/zaimkarta-health-check`. Быстрая проверка выполняется каждые 5 минут и контролирует:

- системный диск и свободное место для бэкапов;
- RAM и load average;
- состояние и неожиданные перезапуски контейнеров `app`, `db`, `caddy`;
- ответы production HTTPS endpoints;
- свежесть локальных DB- и uploads-бэкапов;
- всплески `4xx`, `5xx` и `/go` `429` в Caddy logs.

Глубокая проверка бэкапов запускается ежедневно в `04:32 UTC` после backup cron и сверяет локальные файлы с закрытой копией в Google Drive.

Health endpoint не раскрывает значения ресурсов, IP, логи или детали инфраструктуры. Он возвращает только `200 ok` либо `503 unavailable`, запрещает кеширование и индексацию.

## Результаты проверки

Ручной режим `all` успешно выполнен дважды:

- системный диск: 39%;
- RAM: 15–16%;
- load average: `0.20` при 2 CPU;
- `app` и `db`: healthy;
- `caddy`: running;
- неожиданные перезапуски контейнеров: 0;
- проверяемые HTTPS endpoints: `200`;
- недавние `4xx`, `5xx` и `/go` `429`: 0;
- локальные DB- и uploads-бэкапы свежие;
- локальные файлы совпадают с копиями в Google Drive;
- `/api/health/server`: `200 ok`.

Автоматический пятиминутный health cron подтверждён: endpoint продолжил возвращать `200` после истечения 12 минут от ручного marker, следовательно marker обновился автоматическим запуском.

## Проверка аварийного уведомления

Для безопасной проверки health-монитору временно задали отсутствующий URL. Production-сайт, база, cron и контейнеры не останавливались.

Результат:

1. UptimeRobot зарегистрировал реальный инцидент Down.
2. Аварийное письмо было доставлено на email владельца.
3. Монитору вернули правильный URL `https://zaimkarta.ru/api/health/server`.
4. Монитор восстановился и снова стал зелёным.
5. Recovery-уведомление было доставлено.

Отдельная кнопка Test Notification также проверена; письмо доставлено.

## Рабочие команды на случай проверки

Обычная работа не требует ежедневного входа на VPS. При получении алерта используются:

```bash
sudo /usr/local/sbin/zaimkarta-health-check all
tail -n 100 /var/log/zaimkarta-health.log
curl -i https://zaimkarta.ru/api/health/server
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml ps
```

Если SSH завершается на `banner exchange` или `kex_exchange_identification` до запроса пароля, сначала нужно отключить VPN и сделать одну попытку с обычного домашнего IP. Этот сценарий добавлен в аварийную инструкцию.

## Что осталось вне этапа 8

Следующие пункты не относятся к незавершённому мониторингу и не мешают считать этап 8 закрытым:

- после первого автоматического backup cron проверить `cron.log` и свежую пару файлов в Google Drive — хвост этапа 7;
- безопасно проверить production-сценарии `/go` и uploads;
- разобрать `npm audit` без принудительного обновления зависимостей;
- внедрить 2FA для админки;
- провести финальный pre-launch smoke-test перед включением трафика.

До выполнения этих общих pre-launch пунктов активный рекламный трафик запускать не следует.
