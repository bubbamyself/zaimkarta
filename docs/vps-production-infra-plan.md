# ZaimKarta: план production-инфраструктуры VPS

## Что готовим на этапе 6А

На этом этапе мы готовим локальный «набор для переезда»: пример Docker Compose, конфигурацию Caddy, шаблон переменных окружения и понятный checklist. Эти файлы показывают, как компоненты будут соединены, но сами по себе ничего не публикуют в интернете.

## Чего не делаем на этапе 6А

- не покупаем VPS и доменные услуги;
- не меняем DNS;
- не создаем настоящие production-секреты;
- не открываем порты и не настраиваем SSH, firewall или fail2ban;
- не выпускаем сайт публично и не считаем HTTPS проверенным;
- не настраиваем внешние бэкапы и мониторинг.

Все это делается и проверяется на реальном сервере на этапе 6Б.

## Схема запуска

```text
Интернет
   |
   | 80/443
   v
Caddy (HTTPS, редиректы, логи)
   |
   | внутренняя Docker-сеть, порт 3000
   v
Next.js app
   |
   | закрытая Docker-сеть, порт 5432
   v
PostgreSQL
```

Caddy — это «входная дверь и охранник»: только он смотрит в интернет, принимает HTTP/HTTPS и передает нормальный запрос сайту. Firewall — «список открытых дверей» сервера. PostgreSQL — внутренний «склад»: приложению он нужен, но отдельная дверь со склада прямо на улицу не нужна и опасна.

## Какой VPS покупать для MVP

Разумная стартовая конфигурация:

- Linux Ubuntu 24.04 LTS;
- 2 vCPU;
- 4 ГБ RAM;
- 50–80 ГБ NVMe SSD;
- IPv4-адрес;
- возможность делать snapshot;
- дата-центр, подходящий основной аудитории и требованиям к данным;
- базовая защита провайдера от DDoS.

Минимум 2 ГБ RAM может оказаться тесным: Next.js собирается на сервере после запуска базы и в момент сборки потребляет заметно больше памяти. Если бюджет позволяет, 4 ГБ безопаснее. До покупки нужно проверить правила провайдера, резервные копии, лимиты трафика, доступ к консоли восстановления и порядок удаления данных.

## Порты

Наружу открыты:

- `80/tcp` — нужен для HTTP-редиректа и выпуска HTTPS-сертификата;
- `443/tcp` — основной HTTPS-трафик;
- `443/udp` — HTTP/3, необязателен, но предусмотрен конфигурацией Caddy;
- SSH-порт — только для управления сервером; его номер и ограничения выбираются на этапе 6Б.

Наружу закрыты:

- `3000` — Next.js доступен только Caddy внутри Docker;
- `5432` — PostgreSQL доступен только приложению внутри закрытой Docker-сети;
- все прочие порты, пока для них нет отдельной причины.

Публикации `3000` и `5432` нет в `deploy/docker-compose.prod.example.yml`. Закрытие firewall обязательно отдельно проверить с другого компьютера: Docker-конфигурация не заменяет firewall.

## Почему PostgreSQL нельзя выставлять в интернет

В базе находятся пользователи админки, офферы, клики и служебные данные. Открытый `5432` постоянно сканируют боты, как дверь склада, видимую с улицы. Даже сильный пароль не дает полезного преимущества от такого доступа. Приложение соединяется с базой по внутреннему имени `db`, поэтому внешний порт базе не нужен.

## Зачем нужен Caddy

Caddy автоматически получает и обновляет HTTPS-сертификат, перенаправляет `www` на основной домен, сжимает ответы и пишет access-логи. Для владельца это проще, чем вручную обслуживать сертификаты Nginx.

Ограничение: стандартный Caddy не имеет rate limit без стороннего модуля. Встроенный лимит приложения уже есть, но до публичного запуска `/go/` нужно дополнительно ограничить на уровне провайдера/WAF либо отдельно согласовать Nginx/custom Caddy. Мнимая настройка в example не добавлена.

## Где хранятся логотипы

Приложение пишет загруженные SVG в `/app/public/uploads/logos`. Docker-volume `logo_uploads` сохраняет эту папку при пересоздании контейнера. Без volume новые логотипы исчезали бы вместе со старым контейнером.

Volume защищает от обычного пересоздания контейнера, но не является бэкапом. На этапе 6Б логотипы нужно включить во внешнее резервное копирование вместе с базой.

## Где хранятся секреты

В Git лежит только `deploy/production.env.server.example` с заглушками. На VPS рядом с compose-файлом создается `deploy/production.env.server`, доступ к нему ограничивается владельцем файла, например `chmod 600`. Этот настоящий файл игнорируется Git.

Копия секретов хранится в менеджере паролей. Значения `POSTGRES_PASSWORD`, `ADMIN_SESSION_SECRET` и `LEAD_IP_HASH_SALT` должны быть разными. Пароль внутри `DATABASE_URL` должен совпадать с `POSTGRES_PASSWORD`; специальные символы в URL придется корректно закодировать.

## Как примерно проходит деплой

1. На сервер устанавливаются Docker Engine и Compose plugin.
2. Репозиторий помещается в закрытую рабочую папку deploy-пользователя.
3. Example-файлы копируются без суффикса `.example`.
4. Создается закрытый `production.env.server` с настоящими значениями.
5. DNS домена направляется на VPS.
6. Compose собирает образ приложения и запускает PostgreSQL.
7. После готовности базы контейнер приложения выполняет `prisma migrate deploy`, затем production-сборку Next.js и запуск.
8. Caddy начинает принимать трафик только после успешной healthcheck приложения.

Важно: текущие страницы ZaimKarta читают Prisma-данные во время `next build`. Поэтому сборка выполняется при старте контейнера после готовности базы, а не внутри `docker build`. Первый запуск и каждый пересозданный контейнер будут стартовать медленнее. Это осознанный минимальный вариант для MVP, а не новая архитектура.

## Команды проверки на VPS

Выполнять из папки `deploy/`, после копирования example-файлов и заполнения env:

```bash
docker compose -f docker-compose.prod.yml config
docker compose -f docker-compose.prod.yml build
docker compose -f docker-compose.prod.yml up -d
docker compose -f docker-compose.prod.yml ps
docker compose -f docker-compose.prod.yml logs --tail=200 app caddy db
docker compose -f docker-compose.prod.yml exec app npm exec prisma migrate status
curl -I http://zaimkarta.ru
curl -I https://zaimkarta.ru
curl -I https://www.zaimkarta.ru
```

С другого компьютера проверить, что `80` и `443` доступны, а `3000` и `5432` закрыты. Затем пройти ручной smoke test из `docs/production-launch-checklist.md`: главная, SEO-страницы, админка, региональный фильтр, загрузка логотипа и CPA-переход.

## Что останется на этап 6Б

- выбрать и купить VPS;
- создать отдельного deploy-пользователя и вход по SSH-ключу;
- запретить root/password login по SSH;
- настроить firewall и fail2ban;
- установить Docker из официального источника;
- направить DNS и проверить автоматический HTTPS;
- создать настоящие секреты и admin-пользователя;
- решить proxy-level rate limit для `/go/`;
- настроить ротацию Docker-логов;
- настроить ежедневный `pg_dump` и копию логотипов вне VPS;
- один раз проверить восстановление бэкапа;
- подключить uptime, disk, CPU/RAM и error alerts;
- проверить реальные IP, заголовки, порты и весь production smoke test.

## Стоп-лист: без чего нельзя запускать публично

- DNS и HTTPS не проверены на настоящем домене.
- SSH по паролю/root не закрыт, firewall и fail2ban не настроены.
- `3000` или `5432` доступны из интернета.
- В env остались заглушки, `localhost` или повторно использованные секреты.
- Миграции не применены или production build/healthcheck не проходят.
- В базе есть активные demo CPA-ссылки или ненужные тестовые админы.
- Не решен дополнительный rate limit `/go/` на proxy/provider layer.
- Нет ежедневного внешнего бэкапа базы и логотипов либо не проверено восстановление.
- Нет uptime/disk/error alerts и понятного канала аварийных уведомлений.
