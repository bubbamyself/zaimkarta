# ZaimKarta: MVP security checklist перед production

Документ фиксирует минимальный практический набор мер, после которого ZaimKarta можно выпускать на VPS как MVP. Это не “идеальная корпоративная безопасность”, а разумная защита для первого публичного запуска: закрыть очевидные дыры, не оставить базу и админку наружу, снизить риск ботов, накрутки кликов, потери данных и поломки сервера.

## Краткий вердикт

Запускаться можно после выполнения обязательного checklist из этого документа.

Сейчас проект уже имеет часть правильной базы:

- админская сессия подписывается через `ADMIN_SESSION_SECRET`;
- cookie админки `httpOnly`, `sameSite=lax`, `secure` в production;
- страницы админки помечены как `noindex`;
- CPA-переход `/go/[slug]` делает `302`, проверяет активность оффера, HTTPS CPA-ссылку и региональное ограничение;
- IP для лида хранится как hash, а не открытым текстом;
- загрузка логотипов ограничена SVG, размером 400 КБ, серверной проверкой и безопасным именем файла.

Но перед production обязательно нужно закрыть следующие MVP-риски:

- нет явного rate limit для входа в админку;
- нет явного rate limit и антибот-фильтра для `/go/[slug]`;
- `LEAD_IP_HASH_SALT` сейчас имеет fallback `zaimkarta-local`, в production это недопустимо;
- нет общей проверки production env на `localhost`, слабые секреты и demo CPA-ссылки;
- security headers не настроены в `next.config.ts` или reverse proxy;
- нет описанной и проверенной схемы бэкапов;
- нет мониторинга uptime, диска, CPU/RAM, ошибок приложения и всплесков кликов.

## Оценка текущего research-документа

`zaimkarta-prod-infra-security-research.md` полезен как широкий справочник: там правильно описаны VPS + Docker + Caddy/Nginx, закрытый PostgreSQL, HTTPS, бэкапы, мониторинг, SEO и аварийные сценарии.

Проблема документа для MVP: он слишком широкий. В нем смешаны обязательные меры перед первым релизом и улучшения, которые можно делать позже: staging, WAL/PITR, Prometheus/Grafana, полноценный централизованный логинг, отдельный managed PostgreSQL, WAF/CDN, подробный DR-процесс.

Практическое решение: использовать research как базу знаний, а перед релизом идти по этому MVP-checklist. Если пункт помечен как `обязательно`, без него публичный запуск лучше не делать.

## Что обязательно для MVP

- VPS с закрытым SSH по паролю, отдельным deploy user, firewall и fail2ban.
- Наружу открыты только `80`, `443` и SSH-порт.
- PostgreSQL не доступен из интернета.
- Docker-контейнеры не публикуют лишние порты наружу.
- Caddy или Nginx принимает весь внешний трафик, выдает HTTPS и проксирует в Next.js.
- Production env без `localhost`, demo-секретов и demo CPA-ссылок.
- Сильные `ADMIN_SESSION_SECRET` и `LEAD_IP_HASH_SALT`.
- Secure cookies включены.
- `/admin`, `/api`, `/go` закрыты от индексации.
- Security headers включены хотя бы на базовом уровне.
- Админка имеет сложный пароль и rate limit на login.
- `/go/[slug]` имеет rate limit и базовый bot filter.
- Загрузка логотипов ограничена по размеру, MIME/type, SVG-содержимому и папке хранения.
- Ежедневный `pg_dump` хранится вне VPS, восстановление хотя бы один раз проверено.
- Uptime alert, disk alert, CPU/RAM alert, error alert и канал связи для аварий.

## Что можно позже

- Отдельный staging-сервер.
- Managed PostgreSQL.
- WAL/PITR вместо обычных ежедневных дампов.
- Полноценный WAF/CDN.
- Prometheus + Grafana + Loki.
- 2FA в админке.
- IP allowlist для админки, если работа будет только с фиксированных IP.
- Отдельный admin-поддомен.
- Детальный аудит действий админов.
- Автоматическое распознавание фрода по кликам.

## Checklist перед релизом

- [ ] `NODE_ENV=production`.
- [ ] `NEXT_PUBLIC_SITE_URL=https://zaimkarta.ru`.
- [ ] `DATABASE_URL` указывает на production PostgreSQL, не на локальную базу.
- [ ] `ADMIN_SESSION_SECRET` сгенерирован заново, длинный, уникальный, не хранится в репозитории.
- [ ] `LEAD_IP_HASH_SALT` сгенерирован заново, длинный, уникальный, не хранится в репозитории.
- [ ] В production запрещен fallback `zaimkarta-local` для `LEAD_IP_HASH_SALT`.
- [ ] В `.env`, HTML, canonical, JSON-LD и sitemap нет `localhost`.
- [ ] В базе нет активных demo CPA-ссылок на `example.com` или тестовые URL.
- [ ] Все активные CPA-ссылки начинаются с `https://`.
- [ ] В репозитории нет `.env`, паролей, токенов, приватных ключей и секретов CPA-сетей.
- [ ] GitHub Actions или локально проходят `npm run lint` и `npm run build`.
- [ ] Prisma migrations применены через production-safe команду миграций.
- [ ] Создан только нужный admin-пользователь, лишние тестовые удалены или отключены.
- [ ] Проверен ручной сценарий: главная, SEO-страница, оффер, выбор региона, CTA через `/go/[slug]`, запись клика, запрет перехода при региональном ограничении.

## Checklist настройки VPS

- [ ] Создан отдельный пользователь `deploy`, работа под `root` отключена.
- [ ] SSH-вход только по ключу.
- [ ] `PasswordAuthentication no`.
- [ ] `PermitRootLogin no`.
- [ ] SSH-порт известен владельцу и не конфликтует с firewall.
- [ ] Установлен firewall: разрешены только `80/tcp`, `443/tcp` и SSH.
- [ ] PostgreSQL-порт `5432` не открыт наружу.
- [ ] Docker не публикует наружу порт приложения и базы; наружу смотрит только reverse proxy.
- [ ] Установлен fail2ban для SSH и, при возможности, для Nginx/Caddy.
- [ ] Включены автоматические security updates.
- [ ] Включена ротация логов, чтобы диск не забился логами.
- [ ] У хостера включены базовые DDoS-фильтры или выбран тариф/провайдер с такой защитой.
- [ ] Есть доступ к emergency console у провайдера на случай ошибки SSH/firewall.

## Checklist reverse proxy и HTTPS

- [ ] Используется Caddy или Nginx как единая входная точка.
- [ ] `http://zaimkarta.ru` редиректит на `https://zaimkarta.ru`.
- [ ] `https://www.zaimkarta.ru` редиректит на `https://zaimkarta.ru`.
- [ ] TLS-сертификат получен и автообновляется.
- [ ] Включено gzip/brotli-сжатие.
- [ ] Настроены лимиты размера запроса, чтобы нельзя было заливать большие файлы.
- [ ] Для `/admin/login` и `/go/` на уровне proxy добавлен базовый rate limit, если он еще не реализован в приложении.
- [ ] В access log пишутся IP, URL, статус, user-agent и время ответа.

## Checklist защиты приложения

- [ ] В production нет stack traces для пользователя.
- [ ] `NEXT_PUBLIC_*` переменные не содержат секретов.
- [ ] `ADMIN_SESSION_SECRET` обязателен: без него админка не должна работать.
- [ ] `LEAD_IP_HASH_SALT` обязателен: без него `/go` не должен использовать локальную соль.
- [ ] Cookies для админки и lead tracking имеют `httpOnly`, `secure`, `sameSite=lax`.
- [ ] `/admin/*`, `/api/*`, `/go/*` не попадают в sitemap.
- [ ] `/admin/*`, `/api/*`, `/go/*` закрыты в `robots.txt`.
- [ ] Для admin-страниц есть `robots: { index: false, follow: false }`.
- [ ] Для `/go/[slug]` и `/api/offers/[slug]/click` добавлен `X-Robots-Tag: noindex, nofollow`.
- [ ] Включены базовые security headers: `Strict-Transport-Security`, `X-Content-Type-Options: nosniff`, `Referrer-Policy`, `Permissions-Policy`, `X-Frame-Options` или `frame-ancestors`.
- [ ] Для публичных партнерских ссылок используется `/go/[slug]`, прямые CPA-ссылки не светятся в HTML.
- [ ] Ошибки БД и внутренние ошибки не возвращаются пользователю текстом.

## Checklist защиты админки

- [ ] Пароль администратора длинный, уникальный, хранится в менеджере паролей.
- [ ] Нет admin-пользователей с demo-паролями.
- [ ] При ошибке логина показывается одинаковый текст: “Неверный логин или пароль”.
- [ ] Добавлен rate limit на `/admin/login`: например, 5-10 попыток за 10-15 минут на IP и/или username.
- [ ] После множества ошибок входа создается alert или запись в лог.
- [ ] Cookie сессии `httpOnly`, `secure`, `sameSite=lax`, path `/admin`.
- [ ] Время жизни сессии ограничено. Сейчас 12 часов выглядит нормально для MVP.
- [ ] Logout очищает cookie.
- [ ] Все admin actions проверяют сессию и права, а не только скрывают кнопки в интерфейсе.
- [ ] Экспорт аналитики доступен только админам с правом `analytics`.
- [ ] Решено, нужен ли IP allowlist. Для MVP желательно включить, если владелец работает с 1-2 стабильных IP. Если IP часто меняется, можно отложить, но оставить сильный пароль и rate limit.

## Checklist защиты `/go/[slug]` от накрутки

- [ ] `/go/[slug]` возвращает `302`, не `301`.
- [ ] CPA-секреты и полный tracking URL не показываются на публичной странице.
- [ ] CPA redirect строится только на сервере.
- [ ] Проверяется, что оффер активен.
- [ ] Проверяется, что affiliate offer активен.
- [ ] Проверяется, что CPA-ссылка `https://`.
- [ ] Сервер проверяет региональное ограничение перед редиректом.
- [ ] Добавлен rate limit по IP hash, lead cookie и slug.
- [ ] Подозрительные user-agent, пустые user-agent и очевидные crawlers не отправляются в CPA, а получают fallback или 403/429.
- [ ] Поисковые боты не индексируют `/go/*`.
- [ ] При всплеске кликов по одному slug/IP/user-agent пишется alert.
- [ ] Неуспешные переходы тоже логируются агрегированно: offer not found, restricted region, invalid CPA URL, rate limited.
- [ ] В отчетах для CPA можно отделить нормальные клики от отфильтрованных.

MVP-логика rate limit может быть простой: сначала на уровне Nginx/Caddy, затем в приложении. Для одного VPS достаточно лимита на route `/go/`: например, несколько десятков переходов в минуту на IP и более жесткий лимит на один slug. Точные числа лучше подстроить после первых реальных данных.

## Checklist защиты загрузки логотипов

- [ ] Загрузка доступна только авторизованному администратору с правом управления офферами.
- [ ] Размер ограничен. В коде сейчас `400 КБ`, для MVP нормально.
- [ ] Разрешен только ожидаемый тип. Сейчас это `image/svg+xml`.
- [ ] Проверка выполняется на сервере, не только в браузере.
- [ ] Имя файла генерируется сервером, пользовательское имя не используется.
- [ ] Файлы сохраняются только в `public/uploads/logos`.
- [ ] Нельзя загрузить `.php`, `.js`, `.html`, `.exe` или файл без ожидаемого расширения.
- [ ] SVG проверяется на `<script>`, inline-обработчики `onload/onerror/...` и `javascript:`.
- [ ] Для `/uploads/logos/*` желательно отдавать `Content-Type: image/svg+xml` и `X-Content-Type-Options: nosniff`.
- [ ] Если появятся сомнения по SVG, перейти на PNG/WebP или санитайзить SVG более строго. SVG удобен для логотипов, но это более рискованный формат, чем обычная картинка.

## Checklist бэкапов

- [ ] Каждый день выполняется `pg_dump` production-базы.
- [ ] Бэкап хранится вне VPS: S3-совместимое хранилище, отдельный сервер, облако или локально у владельца.
- [ ] Бэкап не лежит в публичной папке сайта и не доступен по URL.
- [ ] Бэкап сжимается и, желательно, шифруется.
- [ ] Хранение: минимум 7 ежедневных копий и 4 недельные копии.
- [ ] Раз в неделю проверяется, что свежий бэкап действительно создан.
- [ ] До релиза один раз выполнено тестовое восстановление в отдельную базу.
- [ ] Инструкция восстановления хранится отдельно от VPS.

## Минимальный план защиты от DDoS и ботов

- [ ] Выбрать VPS-провайдера с базовой DDoS-защитой.
- [ ] Держать reverse proxy единственной публичной точкой входа.
- [ ] Закрыть все лишние порты.
- [ ] Включить rate limit на `/admin/login`.
- [ ] Включить rate limit на `/go/`.
- [ ] Включить общий лимит запросов на IP для слишком частых запросов к сайту.
- [ ] Ограничить размер body upload.
- [ ] Быстро блокировать IP/подсети через firewall при явной атаке.
- [ ] Иметь заранее подготовленный “режим защиты”: временно усилить лимиты `/go`, закрыть админку по IP allowlist, включить challenge/WAF у провайдера или CDN, если доступно.

Важно: MVP-защита не остановит большую DDoS-атаку уровня оператора связи. Ее задача другая: не дать простому боту, сканеру или накрутчику положить маленький VPS и не дать использовать сайт как бесплатный редирект-автомат.

## Monitoring и аварийная реакция

- [ ] Uptime check главной страницы каждые 1-5 минут.
- [ ] Uptime check одной важной SEO-страницы.
- [ ] Проверка `/go/[slug]` на тестовом/служебном оффере или безопасном fallback-сценарии.
- [ ] Alert по диску: например, больше 80%.
- [ ] Alert по CPU/RAM: длительная высокая нагрузка.
- [ ] Alert по падению контейнеров.
- [ ] Alert по ошибкам приложения через Sentry или аналог.
- [ ] Alert по ошибкам 5xx на reverse proxy.
- [ ] Alert по резкому росту 404/429.
- [ ] Alert по всплескам кликов: общий рост, рост по одному offer slug, рост с одного IP/user-agent.
- [ ] Уведомления приходят в Telegram и/или email владельцу.

### План действий при атаке или поломке

1. Проверить, жив ли VPS и сколько занято CPU/RAM/disk.
2. Посмотреть access log reverse proxy: много ли запросов, какие URL, IP и user-agent.
3. Если бьют `/go`, временно усилить rate limit и отдавать `429`.
4. Если бьют `/admin/login`, временно закрыть `/admin` по IP allowlist.
5. Если забивается диск, остановить рост логов, включить/исправить logrotate, удалить только безопасные старые логи.
6. Если приложение упало после деплоя, откатить контейнер/коммит и проверить миграции.
7. Если есть признаки взлома, изолировать сервер, сменить секреты, пароли, SSH-ключи и CPA-доступы, поднимать чистый сервер из проверенного бэкапа.

- [ ] Добавить production-проверку, что `LEAD_IP_HASH_SALT` задан и не используется `zaimkarta-local`.
- [ ] Добавить rate limit для `/admin/login`.
- [ ] Добавить rate limit для `/go/[slug]`.
- [ ] Добавить `X-Robots-Tag: noindex, nofollow` для `/go/*` и `/api/*`.
- [ ] Добавить или настроить security headers.
- [ ] Подготовить `robots.txt` и убедиться, что `/admin`, `/api`, `/go` закрыты.
- [ ] Проверить, что sitemap содержит только публичные SEO-страницы и offer pages.
- [ ] Настроить VPS по checklist выше.
- [ ] Настроить HTTPS и редиректы домена.
- [ ] Настроить ежедневный `pg_dump` вне VPS и проверить восстановление.
- [ ] Подключить uptime/error/resource alerts.
- [ ] Удалить или выключить demo CPA-ссылки.
- [ ] Создать сильного admin-пользователя и удалить тестовых.

## Задачи после релиза

- [ ] Первые 7 дней ежедневно смотреть логи ошибок, `/go`-клики, 404/429/5xx и загрузку сервера.
- [ ] Проверить индексацию в Яндекс Webmaster и Google Search Console.
- [ ] Проверить, что CPA-сети видят корректные `lead_id`, `click_id`, `sub1`, `sub2`, `sub3`.
- [ ] Проверить, что бэкапы продолжают создаваться каждый день.
- [ ] Добавить журнал входов в админку и важных admin-действий.
- [ ] Подумать об IP allowlist для админки, если станет понятно, откуда владелец обычно заходит.

## Задачи, когда появится трафик

- [ ] Настроить отдельные пороги аномалий по каждому offer slug.
- [ ] Добавить более умную антифрод-логику для `/go`: повторы, слишком быстрые клики, подозрительные user-agent, частые клики по разным офферам.
- [ ] Вынести PostgreSQL в managed DB или хотя бы на отдельный VPS, если база начнет конкурировать за ресурсы с приложением.
- [ ] Добавить WAF/CDN или защиту провайдера повышенного уровня.
- [ ] Настроить полноценные дашборды по кликам, ошибкам, latency, CPU/RAM/disk.
- [ ] Добавить staging-окружение перед крупными изменениями.
- [ ] Рассмотреть 2FA для админки.
- [ ] Перейти от ежедневных дампов к PITR/WAL, если база станет бизнес-критичной.
