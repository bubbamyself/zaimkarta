# ZaimKarta — аварийный план production

Этот документ нужен не для ежедневной работы, а для ситуации «пришёл alert и непонятно, что делать». Главное правило: сначала сохранить данные и понять причину, потом менять сервер.

## Первые пять минут

1. Не паниковать и не выполнять команды из случайной статьи или сообщения.
2. Открыть UptimeRobot и понять, какой монитор красный: весь сайт, отдельный URL или heartbeat сервера.
3. Если админка доступна и ошибка затрагивает посетителей, включить `ТЕХРАБОТЫ`.
4. Подключиться к VPS и выполнить только read-only проверки:

```bash
sudo /usr/local/sbin/zaimkarta-health-check all
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml ps
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml logs --tail=200 app caddy db
df -h
free -h
tail -n 100 /var/log/zaimkarta-health.log
```

5. Записать время начала проблемы и сохранить текст ошибки. Не публиковать логи целиком: в access logs могут быть IP и URL-параметры.

Никогда не выполнять `docker compose down -v`, не удалять Docker volumes и не восстанавливать dump поверх production-базы без отдельного подтверждённого плана.

## Режим `ТЕХРАБОТЫ`

Включить его может `BOSS` или администратор с правом управления администраторами.

Где находится кнопка:

1. Войти в `/admin`.
2. В верхней части внутренней панели нажать `ТЕХРАБОТЫ: ВЫКЛ`.
3. Подтвердить включение.

После включения:

- публичные страницы показывают нейтральную страницу «Технические работы» и закрываются от индексации;
- админка остаётся доступной;
- `/go/[slug]` перестаёт отправлять людей в CPA и возвращает `503` с `Retry-After: 3600`;
- данные и контейнеры не удаляются.

Включать техработы при недоступной базе, массовых 5xx, подозрении на взлом, опасном деплое или перед согласованным восстановлением. Не включать только из-за единичного `404` или одного краткого alert, если сайт уже восстановился.

После включения проверить alert, Docker, диск и бэкап. Выключать режим только после повторной проверки сайта, админки и причины аварии.

## Сайт недоступен

Признаки: UptimeRobot сообщает Down, браузер не открывает главную, несколько URL одновременно недоступны.

Проверить:

```bash
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml ps
curl -I https://zaimkarta.ru/
sudo /usr/local/sbin/zaimkarta-health-check quick
```

Первые безопасные действия: проверить панель Beget, доступность VPS и состояние `caddy`, `app`, `db`. Если VPS полностью недоступен, использовать emergency console провайдера и обратиться к специалисту.

Нельзя перезапускать все контейнеры подряд, менять DNS или переустанавливать сервер, пока не сохранена причина. Техработы полезны, если админка ещё работает; при полном падении VPS включить их через приложение невозможно.

## HTTPS не работает

Признаки: браузер предупреждает о сертификате, HTTP доступен, а HTTPS нет, UptimeRobot сообщает SSL/TLS error.

Проверить `caddy` и его последние логи:

```bash
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml ps caddy
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml logs --tail=200 caddy
```

Проверить, что DNS всё ещё указывает на VPS и порты `80/443` открыты. Не отключать HTTPS и не принимать в браузере предупреждение «всё равно перейти». Если сертификат не обновляется или изменился DNS — обратиться к специалисту.

## Заканчивается диск

Признаки: alert от 80%, ошибки записи, Docker или PostgreSQL ведут себя нестабильно.

Проверить:

```bash
df -h
docker system df
sudo du -xhd1 /var/log /home/deploy/backups 2>/dev/null
```

Сначала определить, что растёт: Docker logs, системные логи, образы или бэкапы. Можно согласованно ротировать старые логи по существующей политике. Нельзя удалять текущие DB/uploads-бэкапы, PostgreSQL volume, `site/public/uploads` или неизвестные Docker volumes. При заполнении выше 90% и продолжающемся росте включить техработы и вызвать специалиста.

## База недоступна

Признаки: `db` unhealthy, приложение выдаёт 5xx, админка и страницы с данными не открываются.

Проверить:

```bash
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml ps db app
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml logs --tail=200 db app
df -h
```

Включить техработы, если админка ещё отвечает. Не выполнять seed, миграцию «наугад», `down -v` или восстановление поверх рабочей базы. Если причина не очевидна либо PostgreSQL сообщает о повреждении данных — прекратить изменения и обратиться к специалисту.

## Контейнер app упал или перезапускается

Признаки: `app` exited/unhealthy, restart count быстро растёт, Caddy возвращает 502/503.

Проверить логи `app`, состояние `db`, свободную память и диск. Один ручной перезапуск `app` допустим только после сохранения ошибки и если база healthy:

```bash
docker compose -f /home/deploy/zaimkarta/deploy/docker-compose.prod.yml restart app
```

Если контейнер снова упал, не повторять перезапуск циклически. Включить техработы, зафиксировать последний деплой/commit и обратиться к разработчику.

## Много `429` на `/go`

Признаки: health-check сообщает о 20 и более `/go` `429` за 10 минут, пользователи жалуются на переходы.

Посмотреть только агрегированные значения:

```bash
sudo /usr/local/sbin/zaimkarta-health-check quick
```

В Caddy logs проверить время, URI и повторяющиеся источники, не публикуя IP и query strings. Рост `429` означает, что rate limit уже отсекает часть запросов; это не обязательно падение сайта. При продолжающемся всплеске включить техработы, временно остановив CPA-переходы, и обратиться к специалисту для provider/WAF rate limit. Не отключать приложение и не увеличивать лимиты вслепую.

## Подозрение на атаку

Признаки: резкий рост `4xx/5xx`, нагрузки, трафика, попыток `/admin/login` или `/go`, неизвестные изменения файлов/аккаунтов.

Первые действия: включить техработы, сохранить логи и snapshot VPS у провайдера, проверить активные SSH-сессии и связаться со специалистом. Не удалять следы, не менять сразу все файлы и не продолжать обычный деплой. Если подтверждён взлом, сервер изолируется, секреты и SSH-ключи меняются, а production поднимается на чистом контуре из проверенного бэкапа.

## Бэкап не создался

Признаки: heartbeat alert, флаг ошибки глубокой проверки, нет свежей пары DB/uploads или Google Drive copy.

Проверить:

```bash
sudo /usr/local/sbin/zaimkarta-backup check
tail -n 100 /home/deploy/backups/zaimkarta/logs/cron.log
tail -n 100 /var/log/zaimkarta-health.log
```

Если рабочий сайт и база исправны, сначала не требуется техработ. Устранить место на диске, доступ Google Drive или ошибку контейнера, затем вручную создать новый бэкап по `docs/production-backup-plan.md` и повторить `check`. Не удалять последнюю успешную копию.

Если свежего проверенного внешнего бэкапа нет более 36 часов, не выполнять рискованный деплой, миграцию или массовое изменение данных.

## Потерян доступ к админке

Признаки: пароль не принимается, сессия сброшена, аккаунт заблокирован rate limit.

Сначала подождать указанное время блокировки и проверить правильный адрес `/admin/login`. Не создавать нового администратора через seed и не менять данные напрямую в базе. Если пароль действительно потерян, использовать отдельно согласованную процедуру создания/сброса production-администратора и проверить, что старый доступ отозван.

## Нужно восстановление из бэкапа

Восстановление — не первая реакция на обычное падение контейнера. Оно нужно только при подтверждённой потере или повреждении данных.

Порядок:

1. Включить техработы.
2. Остановить новые изменения данных.
3. Сохранить snapshot текущего аварийного диска/volume у провайдера.
4. Выбрать последнюю проверенную внешнюю копию.
5. Выполнить `sudo /usr/local/sbin/zaimkarta-backup restore-test` во временном изолированном контейнере.
6. Проверить шесть ключевых таблиц и архив логотипов.
7. Подготовить отдельный план замены production-данных и получить подтверждение владельца.
8. Только после этого восстанавливать production и проходить полный smoke test.

Подробности находятся в `docs/production-backup-plan.md`. Никогда не направлять `pg_restore` прямо в production-базу «для проверки».

## Когда обязательно нужен специалист

- VPS не загружается или недоступна emergency console;
- подозрение на взлом;
- повреждение PostgreSQL или файловой системы;
- потеря данных;
- неизвестные процессы, пользователи или SSH-ключи;
- сайт продолжает падать после одного осознанного перезапуска;
- требуется восстановление production;
- нужно менять firewall, DNS, Docker volumes или секреты в аварийной обстановке.
